# 决策日志

## 2026-07-18 · 速听页操作后置，课程页去重并保持播放可见

- 背景：资讯速听页把视频课堂、历史记录和导入三个操作放在顶部，手机端压缩了批次信息；课程页右栏重复展示一份完整知识地图，且手机向下阅读深讲时视频离开视口。
- 决策：三个速听操作统一移到正文末尾；课程页删除右栏“二·知识地图”表格，但保留视频旁的时间戳跳转导航；手机端仅在视频正在播放且原位置滚出顶部后启用悬浮播放器，暂停时恢复原位。桌面继续沿用左栏吸附。
- 理由：页头只承担身份和当前批次信息，低频导航不应挤占手机首屏；知识地图只保留一个可操作版本即可；播放画面持续可见能让用户边看深讲边核对原片，又不在暂停阅读时长期遮挡内容。

## 2026-07-18 · 默认首页恢复原版，学习工作台只保留为备选稿

- 背景：学习工作台已经按“资讯速听、视频课程、我的笔记、最近记录”落地，但用户最终判断新版仍不适合日常使用，明确要求不再改版并切回原页面。
- 决策：`/prototype/` 恢复到改版前已使用的页面；学习工作台另存为 `/prototype/learning-home.html`，不再作为当前产品定义或默认导航。
- 理由：默认入口应以用户实际愿意持续使用为准，而不是以改版投入或概念完整度为准。保留稿只用于未来参考，不影响现有流程。

## 2026-07-18 · 新首页采用第一版视觉与第三版布局，不新增知识库

- 背景：用户在三套知识工作区方案中选择第一张的克制清爽视觉语言，同时更偏好第三张以课程笔记、关联概念、待复习和最近记录为主的内容布局；并指出当前产品并没有“我的学习”或“知识库”。
- 决策：`/prototype/` 新增个人学习聚合首页，视觉按第一张方案，内容层级按第三张方案；“我的学习”只作为首页标题和聚合视图，不包装成新的后台能力。导航沿用真实入口，不设置“知识库”。完整知识库仍保持在 MVP 范围之外。
- 兼容：现在线上 `/prototype/index.html` 在替换前保存为同目录独立可访问的旧版页面，确保后续可直接回看或恢复；新版不删除原课程、资讯速听和历史记录入口。
- 信息优先级：资讯速听与视频课程是首页两个高频主体；资讯按最新一期与往期批次组织，课程展示真实继续状态；学习后的个人笔记单独沉淀，最近记录统一收纳跨类型活动。没有独立知识库时不展示待复习队列。

## 2026-07-17 · 所有模型管线先过预算与内容指纹，再允许调用

- 背景：Gmail 任务原先对时间窗口内每封邮件逐封调用模型，最后才截取 10 条；课程与中文翻译也没有统一 usage 账本。多会话持续迭代后，页面和自动化进度已经超过真实价值验证，重复生成与失败重试的 Token 无法完整追踪。
- 决策：Gmail 在创建模型客户端前完成明显无效内容过滤、内容去重、候选数量限制及总调用数/字符/估算 Token 预算；超限时零模型调用。邮件、课程和中文翻译统一以“输入内容＋模型＋prompt/schema 版本”作为缓存键，每次尝试无论成功、校验失败或最终失败都写入 Git 忽略的本地 usage 账本。
- 理由：成本控制必须发生在调用之前，最终输出条数不能代表实际消耗；内容指纹同时解决相邻定时窗口重复邮件、同视频重跑和局部翻译重跑。完整账本是后续收紧输出上限和评估真实单位成本的前提。
- 边界：Token 预估按中英文混合文本保守估算，只用于调用前守门，不冒充供应商账单；具体价格和长期预算必须等 3—5 次真实 usage 样本后再决定。当前仍保持手工触发，不因具备守门机制就宣称自动任务或产品价值已验证。

## 2026-07-15 · 资讯速听默认使用固定暖声线音频

- 背景：首个公网原型没有随样例发布 MP3，默认落入浏览器 `speechSynthesis`；实际手机试听明显机械，而且不同系统和浏览器声线不一致。
- 决策：公网样例改用固定生成的 `zh-CN-XiaoxiaoNeural / +4% / -2Hz` 逐句 MP3，文字与音频继续共用同一脚本；系统语音只作为用户导入内容缺失音频时的显式降级。
- 理由：快速资讯是低视觉占用、连续收听场景，跨设备一致性和低疲劳比零音频体积更重要；暖声线比新闻播报腔更符合既定的自然、松弛、单人快讲方向。
- 待验证：用 8—10 分钟真实资讯批次连续收听，确认专有名词、句间衔接、疲劳度和 1.15× 默认播放速度；当前四条样例只能验证跨设备一致，不能证明长听通过。

## 2026-07-16 · 连续资讯采用开场、报幕、正文三阶段编排

- 背景：多条短资讯直接拼接时，纯听场景缺少视觉提示，用户无法稳定识别换题；邮件导入后如果只看到批次生成时间，还会把旧内容误认为新资讯。
- 决策：整批播放先播一段“条数＋主要主题”的开场；第一条直接进入正文，之后每条正文前播放包含序号与主题的报幕。用户手动选择某条或某句时直接播放正文。所有日期提示取来源 `published_at`，批次 `generated_at` 只表示整理时间。
- 理由：开场建立预期，报幕恢复资讯边界，来源日期帮助用户判断时效；三者都复用文字与语音同源脚本，不增加新的内容形态。
- 边界：来源未提供日期时显示“日期未提供”，不从正文猜测；固定声线音频需在用户明确同意发送邮件派生摘要给外部 TTS 后生成，未授权时保持明确标记的设备语音降级。

## 2026-07-16 · 资讯短讲先建立上下文，再给结论

- 背景：首批邮件短讲主要采用“高密度事实＋系统判断”，已经熟悉主题的人可以快速扫结论，但初听者不知道主体是谁、内容在回应什么问题，也难以理解观点背后的原理或趋势。
- 决策：单条默认采用“主体与背景 → 内容与主张 → 原理或趋势 → 判断与限定”的讲述顺序。新闻、模型、观点、产品和案例可按类型调整，但听完必须能复述“谁在讲、这是什么、传递了什么”。
- 理由：资讯速听的目标是快速形成可靠的最小理解，不是追求最高压缩率。允许为上下文增加时长，9 条批次上限放宽到 15 分钟；事实、来源观点、AI 解释和 AI 判断仍需分开并可追溯。
- 待验证：连续听完后能否闭卷复述至少 4/5 条的主体、内容和一个观点；新增上下文是否减少回看文字或倒退播放。

## 2026-07-16 · Gmail 定时任务本机运行，私有简报同步个人服务器

- 背景：手工 Gmail 批次已验证读取、去重、四段式讲述和连续音频，用户希望配置第三方模型 Key、按时间自动读取，并能在手机收听。源码仓库与 GitHub Pages 均为公开，不能承载私人邮件派生产物。
- 决策：定时任务在用户 Mac 上运行，凭据从 Git 忽略的 `.env` 读取；默认使用 DeepSeek 官方 OpenAI 兼容接口和 `deepseek-v4-flash`，按单封邮件生成候选条目，本地完成去重、Pydantic 和段落引用校验，再生成固定音频。只有播放器、结构化摘要和音频同步到启用 Basic Auth 的腾讯云个人站点；公网 HTTPS 入口在 443 或命名 Tunnel 就绪后启用。
- 理由：本机持有 Gmail 与模型密钥，公开 Git 只保存代码和模板；每次生成独立批次，失败不覆盖上一批。手机可访问最终简报，同时避免将邮件正文或密钥放入 GitHub Actions、公开 Pages 或服务器。
- 边界：邮件正文会发送给用户选择的 DeepSeek API，最终脚本会发送给 Edge TTS；腾讯云保存摘要、来源 Gmail 链接和音频，不保存邮件全文。免账户 Tunnel、未备案域名和需要安装客户端的私有网络方案均不作为正式入口；正式入口使用 HTTPS 与登录态保护页面、JSON 和音频，会话密钥仅保存在服务器。具体公网地址、证书有效期和运行状态属于部署运维信息，不写入长期产品决策；后续可平滑切换到已备案域名。

## 2026-07-15 · 新增多源资讯速听作为课程前的快速判断层

- 背景：用户每天面对 AI 新闻、产品发布、模型更新、X 帖、邮件订阅、文章、播客和视频，逐条完整阅读或收听成本过高；现有产品只处理单条长视频的课程化，无法先筛选哪些内容值得投入。
- 决策：在同一产品中新增“资讯速听/快速判断层”，不拆成独立产品，也不新增课程模板。它接收用户手工投递的文字、链接、邮件文件、图片、PDF 和音视频，把每条内容压缩为 20—60 秒、复杂项最多 90 秒的单人语音与同源文字；连续播放时允许跳过、收藏、打开原文、查证和深挖。长视频只有被选中后才进入既有课程导读和深讲流程。
- 内容边界：每条必须区分事实、作者观点和 AI 判断，并绑定文章段落、PDF 页码、图片区域、邮件位置或音视频时间戳。无法安全保留关键限定时标记“建议深读”，不强行压缩；资讯速听只证明判断效率，不证明机制理解或知识迁移。
- 首版范围：只做手工粘贴、公开链接和本地文件导入；邮件以复制正文或 `.eml` 导入。Gmail、RSS、X 账号授权、自动聚合、黑箱个性化、双主播播客、默认外部搜索和完整知识库后置。
- 语音选择：默认单人、高密度、自然松弛且非新闻播报腔；文字与语音共用同一脚本。理由是快速判断要求信息密度，双主播对话会引入不必要的重复与填充，项目既有试听也证明“专业新闻腔”不等于长听舒适。
- 理由：现有市场分别解决了多源收件箱、摘要、音频概览或新闻语音流，但仍缺少“用户自己的多源输入 → 保留单条边界的短语音队列 → 句级溯源 → 一键深挖”的完整闭环。该层可以复用现有提取、转录、TTS、播放器和时间戳能力，同时保持课程产品的深度学习边界。
- 待验证：10 条批次能否在 8 分钟内完成判断；与完整消费后的兴趣选择是否达到 80% 一致且无“必看”漏判；不同来源的证据锚点是否稳定；连续 8—10 分钟单声线是否低疲劳；手工投递通过后哪些自动连接器最有价值。

## 2026-07-13 · 访谈首版用同一自然声线做参数化角色区分

- 背景：`Yunyang/Yunjian` 的真实连续试听偏新闻播报，过于严肃；默认 `Yunxi` 又偏年轻且单声线无法辨别角色。
- 决策：主持人使用 `zh-CN-YunxiNeural / -4Hz / +2%`，嘉宾使用 `zh-CN-YunxiNeural / -10Hz / -2%`；manifest 保存角色映射，每条 cue 保存 `speaker_id`，本地覆盖文件可修正自动识别结果。
- 理由：保留用户已确认的松弛听感，通过适度音高与语速差异区分角色，不引入新的 TTS 供应商或情绪模型。
- 待验证：用同一段 10 分钟访谈确认角色可辨、长听不累；Edge TTS 不支持精细对话韵律，若仍机械，再评估对话型 TTS，而不是继续堆叠音高参数。

## 2026-07-13 · 中文播放采用原片同步翻译音轨

- 背景：中文字幕能跨过语言障碍，但用户必须持续阅读，容易错过代码、图表和产品演示，也无法在低视觉占用场景学习。
- 决策：在原片模式中增加中文音轨。它按原视频顺序翻译，以 3—12 秒语义句同步原画；不补充解释，不做逐字或口型对齐。用户可在同一时间点切回原声核对。中文助讲继续处理认知断点，AI 重讲继续负责教学重组。
- 理由：语言替换、补充解释和重建课程是三种不同学习任务；分轨后用户能明确知道自己听到的是作者内容还是 AI 教学内容，同时复用现有时间戳与知识地图。
- 开源边界：用户克隆仓库后，必须能用自己的本地环境完成第三方视频的字幕获取、翻译、TTS 和播放；这是开源版的核心能力。仓库发布管线、schema、播放器和自由许可 fixture，用户生成的字幕与中文音轨进入 gitignore 目录，不随仓库或公共课程包分发，也不由项目方服务器处理。公开视频仍通过 YouTube 官方嵌入播放器展示。
- 待验证：中文语速与局部画面降速的取舍、访谈多声线价值、关键音效是否需要授权后的混音、四类视频的同步通过率。详见 `docs/中文播放模式方案.md`。

## 2026-07-12 · 自动课程生成先做本地静态管线

- 背景：GitHub Pages 无后端，不能安全持有模型密钥；当前版权路线也是个人本地学习工具。
- 决策：自动生成在本地 CLI 完成，使用 yt-dlp 获取字幕、OpenAI Responses API 结构化生成、Pydantic 校验，输出可由 GitHub Pages 托管的静态 JSON 与通用课程页。
- 理由：密钥不进入前端；生成与观看解耦；失败的模型输出不会污染已发布课程；仍保留时间戳追溯。
- 边界：首版只支持 OpenAI，不建立多供应商抽象；没有 `OPENAI_API_KEY` 时只能运行 fixture 和确定性环节测试。
- 待验证：第二条真实视频能否无需手工改 HTML 生成合格课程；单次 Token 成本和生成耗时。

## 2026-07-12 · 首课验收收缩为闭卷复述

- 背景：原测量协议同时要求决策、时间戳抽查、计时和复述，首轮体验负担过高。
- 决策：Codex 访谈首课只做一项真实验收——学完后闭卷复述“实现廉价后产品工作为何倒转”，提交后对照四个机制要点。
- 理由：先验证课程是否帮助形成可复述的机制模型；减少测试流程本身对体验的干扰。
- 边界：这一结果不能证明学习时间下降或全片时间戳准确；这些指标留到多样例阶段再测。

## 2026-07-12 · 重讲模式技术路线：自建分镜管线为主线

- 背景：AI 教师重讲模式有三条候选实现——NotebookLM 视频概览（托管、成熟、零构建）、自建"脚本＋TTS＋HTML 分镜"管线、HyperFrames（HeyGen 开源 HTML→MP4 渲染引擎）直出成片。
- 决策：自建管线为主线（重讲课作为课程页的一种播放模式）；NotebookLM 只作基线实验，验证"重讲形态值不值得做"；HyperFrames 作为导出层，把分镜 HTML 渲染成可分享的 MP4。
- 理由：可溯源（时间戳回原片）与可交互（追问/换深度）是产品核心差异，托管生成和 MP4 终态都无法满足；HyperFrames 与分镜 HTML 技术栈零摩擦，使"网页课"与"成片"共用一条内容管线。
- 待验证：NotebookLM 基线实验的复述质量对比；分镜 schema 是否足以支撑 HyperFrames 渲染；详见 `docs/重讲模式方案.md` 2.1。

## 2026-07-11 · 学习课程，而非英语工具

- 背景：原始痛点包含英文字幕和英文逐字稿，但用户真正希望更快理解 AI 技术内容。
- 决策：课程学习是主线；默认中文讲解，英文讲解只是可切换选项。

## 2026-07-11 · 默认用户是进阶中的 vibe coding 开发者

- 背景：完全小白式解释会冗长，专家式压缩讨论又难以吸收。
- 决策：默认用户有基础项目经验，能借助 AI 开发，但需要补齐系统工程理解。

## 2026-07-11 · 先验证课程结构，后做数字人

- 背景：完整讲解视频很有吸引力，但视频生成成本和实现复杂度较高。
- 决策：MVP 先做结构化课程与中文音频讲解，数字人和复杂视频剪辑后置。
- 理由：真正需要先验证的是解释、案例和实践能否提高理解效率，而不是讲解人的外观。

## 2026-07-11 · 所有解释必须可追溯

- 背景：技术内容中，AI 容易把作者观点、归纳和外部知识混在一起。
- 决策：明确标注作者原话、AI 归纳、AI 解释和外部补充；视频内结论附时间戳。
- 理由：用户需要能够核对原始上下文，并判断观点归属。

## 2026-07-11 · 章节不等于讲解单元

- 背景：长视频中只有少数内容值得额外讲解，逐章插入解释会让课程更长、更烦。
- 决策：每个语义段落先被分为深讲、快讲、原片、摘要或跳过；用户可关闭助讲或覆盖推荐。
- 理由：课程价值来自选择和编排，不是覆盖率。
- 待验证：五项讲解价值评分在不同视频类型中的权重。

## 2026-07-11 · 使用多模板课程编排

- 背景：工程教程、模型解读、产品评测和访谈的知识结构与最佳视觉表达不同。
- 决策：首版建立四个主模板，混合视频允许章节级选择模板。
- 理由：统一流程可以共享，但讲解重点、证据标准和呈现方式不能统一套壳。
- 待验证：真实视频是否还需要增加“现场编码/直播”和“新闻综述”模板。

## 2026-07-11 · 版权与 ToS 路线：先做个人本地学习工具

- 背景：产品涉及下载音频转录（触碰 YouTube ToS）、保留原片片段插入讲解（对他人视频的再分发与演绎）、基于原视频内容重讲生成课程（独创性归属模糊）。这是产品对外化的最大合规风险，此前无任何决策。
- 决策：验证阶段按"个人本地学习工具"路线执行——本地运行、仅自用、不分发任何含原片素材的产物。对外产品化是独立决策，届时必须重新评估：助讲版能否只以"浏览器插件叠加层/时间戳跳转"形式实现而不复制原片；重讲版是否只用自产讲解素材。
- 理由：自用学习不阻塞验证；而合规路线会反向决定技术架构（本地 CLI vs 在线服务），不能在架构确定后再补。
- 待验证：对外路线下，官方字幕 API、创作者授权合作、纯跳转不复制三种方案的可行性。

## 2026-07-11 · 成功指标必须有测量协议

- 背景：初版成功指标全部是"用户能否……"的定性问句，无法判断黄金测试是否通过。
- 决策：每条指标配测量方式与通过标准，写入 `docs/需求与方案.md` 第 6 节；对照基线为"裸看同类视频"的耗时与复述质量。
- 理由：验证阶段用户是本人，主观感受极易自我说服；没有协议，测试结果不可信也不可比。
- 待验证：1/3 时间目标是否现实，首轮先以 1/2 为通过线。

## 2026-07-11 · 互动课程页是主形态

- 背景：文字、音频和视频各有优势，完整视频生成成本最高且不便于跳转追问。
- 决策：互动课程页作为学习母体；图文、音频和视频从同一课程结构派生。
- 理由：它最适合按需深入、跳过讲解、原文追溯和案例展开，也能复用中间产物控制成本。
# 2026-07-18 · 首页视觉结构以“个人学习书架”为后续探索基线

- 背景：两轮首页视觉探索后，用户明确选择第二轮第二张作为继续迭代基础，并要求只比较不同色系与更有质感的材质风格。
- 选择：后续首页视觉探索先固定“问候＋快速生成＋继续学习＋最近生成＋资讯速听”的个人学习书架结构，不再回到官网式价值宣言和功能陈列。
- 理由：该结构更像打开即用的个人学习产品，既能突出当前课程，又能自然容纳生成入口和已有学习资产。
- 待确认：最终色系与材质尚未选定；本轮比较深绿暖木、清透矿物蓝和紫棕琥珀夜间三种方向，选择后再进入实现。

## 2026-07-18 · 首页以个人真实状态代替产品步骤宣讲

- 背景：用户进一步确认首页主要供个人长期使用，未来开源也不意味着首页需要承担对外营销；现有概念仍把四个课程模块表现成显眼的步骤流程。
- 决策：首页主内容改为历史/收听记录、当前视频学习进度和知识点状态；导读、知识地图、深讲、迁移实践只在当前课程上下文中表达完成度，不作为“第一步到第四步”的功能介绍。新视频生成入口保留，但不压过继续学习。
- 理由：返回用户最需要回答的是“我上次到哪、现在继续什么、哪些还没掌握”，不是再次理解产品能力。真实状态也更适合个人工具与开源自托管场景。
- 视觉边界：在选中的个人学习书架上使用亮、清、透明的彩色叠片和边缘折射；避免灰白磨砂玻璃层层套叠，透明材质不能牺牲文字对比或把内容切成卡片墙。

## 2026-07-18 · 首页采用知识工作区隐喻，视频退回来源位置

- 背景：清透状态版虽然展示了进度和知识状态，但仍以大视频画面、播放按钮和时间轴作为首屏视觉中心，用户感知为“播放器”；彩色亚克力装饰也削弱了学习工具的可信感。
- 决策：首页改用主流知识工作区的组织方式：稳定侧栏或移动端顶级导航、页面式主内容、可扫描的学习条目和知识状态视图。视频缩略图与播放位置只作为课程来源属性，不能主导构图。
- 理由：产品价值是把来源转成可复习、可追溯、可迁移的知识，而不是改善视频消费。知识工作区更容易承载课程页、笔记、概念状态、历史活动和继续学习。
- 参考边界：借鉴 Notion、Craft、Readwise 等产品的克制排版、块/页面组织、列表与属性表达，不复制品牌、图标、文案或具体界面；避免模板化个人仪表盘、播放器大封面和装饰性透明材质。

## 2026-07-18 · 手机课程阅读只吸附学习上下文

- 背景：手机实测中页面吸附了全局页头，播放中的视频没有稳定跟随；正文时间戳和长模块标题也分别造成视觉干扰与阅读迷路。
- 决策：1020px 以下取消全局页头吸附；视频仅在播放或缓冲且滚出原位后悬浮，暂停或结束立即复位；当前深讲模块标题在模块范围内吸附，并在视频悬浮时自动位于视频下方。正文时间戳默认使用低对比浅色，只在交互时强调。
- 理由：手机长文阅读需要持续保留的是当前视频与当前观点，而不是占据大量高度的全局导航；时间戳是溯源入口，不应与正文争夺注意力。

## 2026-07-18 · 跨 App 播放交给 YouTube 与手机系统画中画

- 背景：用户希望离开课程网页、切换到其他手机 App 后仍保留视频窗口。当前课程使用跨域 YouTube IFrame Player，网页只能控制官方 API 暴露的播放、暂停、跳转和状态，不能取得 iframe 内部 `HTMLVideoElement` 来强制调用 Web PiP。
- 决策：课程页显式启用 `playsinline`，保留页内滚动悬浮；另提供低权重“在 YouTube 中打开”入口，让支持的 iOS/Android 与 YouTube 账户通过原生画中画接管跨 App 播放。不提供一个无法保证生效的网页“开启画中画”按钮。
- 边界：画中画是否可用取决于设备系统开关、YouTube 设置、账户/地区和内容类型；网页不能承诺所有手机都能跨 App 继续播放。
