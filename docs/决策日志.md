# 决策日志

## 2026-07-15 · 资讯速听默认使用固定暖声线音频

- 背景：首个公网原型没有随样例发布 MP3，默认落入浏览器 `speechSynthesis`；实际手机试听明显机械，而且不同系统和浏览器声线不一致。
- 决策：公网样例改用固定生成的 `zh-CN-XiaoxiaoNeural / +4% / -2Hz` 逐句 MP3，文字与音频继续共用同一脚本；系统语音只作为用户导入内容缺失音频时的显式降级。
- 理由：快速资讯是低视觉占用、连续收听场景，跨设备一致性和低疲劳比零音频体积更重要；暖声线比新闻播报腔更符合既定的自然、松弛、单人快讲方向。
- 待验证：用 8—10 分钟真实资讯批次连续收听，确认专有名词、句间衔接、疲劳度和 1.15× 默认播放速度；当前四条样例只能验证跨设备一致，不能证明长听通过。

## 2026-07-15 · 新增多源资讯速听作为课程前的快速判断层

- 背景：用户每天面对 AI 新闻、产品发布、模型更新、X 帖、邮件订阅、文章、播客和视频，逐条完整阅读或收听成本过高；现有产品只处理单条长视频的课程化，无法先筛选哪些内容值得投入。
- 决策：在同一产品中新增“资讯速听/快速判断层”，不拆成独立产品，也不新增课程模板。它接收用户手工投递的文字、链接、邮件文件、图片、PDF 和音视频，把每条内容压缩为 20—60 秒、复杂项最多 90 秒的单人语音与同源文字；连续播放时允许跳过、收藏、打开原文、查证和深挖。长视频只有被选中后才进入既有课程导读和深讲流程。
- 内容边界：每条必须区分事实、作者观点和 AI 判断，并绑定文章段落、PDF 页码、图片区域、邮件位置或音视频时间戳。无法安全保留关键限定时标记“建议深读”，不强行压缩；资讯速听只证明判断效率，不证明机制理解或知识迁移。
- 首版范围：只做手工粘贴、公开链接和本地文件导入；邮件以复制正文或 `.eml` 导入。Gmail、RSS、X 账号授权、自动聚合、黑箱个性化、双主播播客、默认外部搜索和完整知识库后置。
- 语音选择：默认单人、高密度、自然松弛且非新闻播报腔；文字与语音共用同一脚本。理由是快速判断要求信息密度，双主播对话会引入不必要的重复与填充，项目既有试听也证明“专业新闻腔”不等于长听舒适。
- 理由：现有市场分别解决了多源收件箱、摘要、音频概览或新闻语音流，但仍缺少“用户自己的多源输入 → 保留单条边界的短语音队列 → 句级溯源 → 一键深挖”的完整闭环。该层可以复用现有提取、转录、TTS、播放器和时间戳能力，同时保持课程产品的深度学习边界。
- 待验证：10 条批次能否在 8 分钟内完成判断；与完整消费后的兴趣选择是否达到 80% 一致且无“必看”漏判；不同来源的证据锚点是否稳定；连续 8—10 分钟单声线是否低疲劳；手工投递通过后哪些自动连接器最有价值。

## 2026-07-13 · 访谈首版用同一自然声线做参数化角色区分

- 背景：`Yunyang/Yunjian` 的真实连续试听偏新闻播报，过于严肃；默认 `Yunxi` 又偏年轻且单声线无法辨别角色。
- 决策：主持人使用 `zh-CN-YunxiNeural / -4Hz / +2%`，嘉宾使用 `zh-CN-YunxiNeural / -10Hz / -2%`；manifest 保存角色映射，每条 cue 保存 `speaker_id`，本地覆盖文件可修正自动识别结果。
- 理由：保留用户已确认的松弛听感，通过适度音高与语速差异区分角色，不引入新的 TTS 供应商或情绪模型。
- 待验证：用同一段 10 分钟访谈确认角色可辨、长听不累；Edge TTS 不支持精细对话韵律，若仍机械，再评估对话型 TTS，而不是继续堆叠音高参数。

## 2026-07-13 · 中文播放采用原片同步翻译音轨

- 背景：中文字幕能跨过语言障碍，但用户必须持续阅读，容易错过代码、图表和产品演示，也无法在低视觉占用场景学习。
- 决策：在原片模式中增加中文音轨。它按原视频顺序翻译，以 3—12 秒语义句同步原画；不补充解释，不做逐字或口型对齐。用户可在同一时间点切回原声核对。中文助讲继续处理认知断点，AI 重讲继续负责教学重组。
- 理由：语言替换、补充解释和重建课程是三种不同学习任务；分轨后用户能明确知道自己听到的是作者内容还是 AI 教学内容，同时复用现有时间戳与知识地图。
- 开源边界：用户克隆仓库后，必须能用自己的本地环境完成第三方视频的字幕获取、翻译、TTS 和播放；这是开源版的核心能力。仓库发布管线、schema、播放器和自由许可 fixture，用户生成的字幕与中文音轨进入 gitignore 目录，不随仓库或公共课程包分发，也不由项目方服务器处理。公开视频仍通过 YouTube 官方嵌入播放器展示。
- 待验证：中文语速与局部画面降速的取舍、访谈多声线价值、关键音效是否需要授权后的混音、四类视频的同步通过率。详见 `docs/中文播放模式方案.md`。

## 2026-07-12 · 自动课程生成先做本地静态管线

- 背景：GitHub Pages 无后端，不能安全持有模型密钥；当前版权路线也是个人本地学习工具。
- 决策：自动生成在本地 CLI 完成，使用 yt-dlp 获取字幕、OpenAI Responses API 结构化生成、Pydantic 校验，输出可由 GitHub Pages 托管的静态 JSON 与通用课程页。
- 理由：密钥不进入前端；生成与观看解耦；失败的模型输出不会污染已发布课程；仍保留时间戳追溯。
- 边界：首版只支持 OpenAI，不建立多供应商抽象；没有 `OPENAI_API_KEY` 时只能运行 fixture 和确定性环节测试。
- 待验证：第二条真实视频能否无需手工改 HTML 生成合格课程；单次 Token 成本和生成耗时。

## 2026-07-12 · 首课验收收缩为闭卷复述

- 背景：原测量协议同时要求决策、时间戳抽查、计时和复述，首轮体验负担过高。
- 决策：Codex 访谈首课只做一项真实验收——学完后闭卷复述“实现廉价后产品工作为何倒转”，提交后对照四个机制要点。
- 理由：先验证课程是否帮助形成可复述的机制模型；减少测试流程本身对体验的干扰。
- 边界：这一结果不能证明学习时间下降或全片时间戳准确；这些指标留到多样例阶段再测。

## 2026-07-12 · 重讲模式技术路线：自建分镜管线为主线

- 背景：AI 教师重讲模式有三条候选实现——NotebookLM 视频概览（托管、成熟、零构建）、自建"脚本＋TTS＋HTML 分镜"管线、HyperFrames（HeyGen 开源 HTML→MP4 渲染引擎）直出成片。
- 决策：自建管线为主线（重讲课作为课程页的一种播放模式）；NotebookLM 只作基线实验，验证"重讲形态值不值得做"；HyperFrames 作为导出层，把分镜 HTML 渲染成可分享的 MP4。
- 理由：可溯源（时间戳回原片）与可交互（追问/换深度）是产品核心差异，托管生成和 MP4 终态都无法满足；HyperFrames 与分镜 HTML 技术栈零摩擦，使"网页课"与"成片"共用一条内容管线。
- 待验证：NotebookLM 基线实验的复述质量对比；分镜 schema 是否足以支撑 HyperFrames 渲染；详见 `docs/重讲模式方案.md` 2.1。

## 2026-07-11 · 学习课程，而非英语工具

- 背景：原始痛点包含英文字幕和英文逐字稿，但用户真正希望更快理解 AI 技术内容。
- 决策：课程学习是主线；默认中文讲解，英文讲解只是可切换选项。
- 理由：语言只是内容获取障碍，知识结构、解释、案例和实践才决定学习效果。
- 排除：不单独设计词汇、语法、听力训练和英语学习进度。

## 2026-07-11 · 默认用户是进阶中的 vibe coding 开发者

- 背景：完全小白式解释会冗长，专家式压缩讨论又难以吸收。
- 决策：默认用户有基础项目经验，能借助 AI 开发，但需要补齐系统工程理解。
- 讲解原则：通俗但不幼稚；重点解释为什么、怎么工作、如何取舍、什么时候失效、如何实践。
- 待验证：是否需要让用户选择“基础 / 进阶 / 深入”三档，还是通过追问动态适配即可。

## 2026-07-11 · 先验证课程结构，后做数字人

- 背景：完整讲解视频很有吸引力，但视频生成成本和实现复杂度较高。
- 决策：MVP 先做结构化课程与中文音频讲解，数字人和复杂视频剪辑后置。
- 理由：真正需要先验证的是解释、案例和实践能否提高理解效率，而不是讲解人的外观。

## 2026-07-11 · 所有解释必须可追溯

- 背景：技术内容中，AI 容易把作者观点、归纳和外部知识混在一起。
- 决策：明确标注作者原话、AI 归纳、AI 解释和外部补充；视频内结论附时间戳。
- 理由：用户需要能够核对原始上下文，并判断观点归属。

## 2026-07-11 · 章节不等于讲解单元

- 背景：长视频中只有少数内容值得额外讲解，逐章插入解释会让课程更长、更烦。
- 决策：每个语义段落先被分为深讲、快讲、原片、摘要或跳过；用户可关闭助讲或覆盖推荐。
- 理由：课程价值来自选择和编排，不是覆盖率。
- 待验证：五项讲解价值评分在不同视频类型中的权重。

## 2026-07-11 · 使用多模板课程编排

- 背景：工程教程、模型解读、产品评测和访谈的知识结构与最佳视觉表达不同。
- 决策：首版建立四个主模板，混合视频允许章节级选择模板。
- 理由：统一流程可以共享，但讲解重点、证据标准和呈现方式不能统一套壳。
- 待验证：真实视频是否还需要增加“现场编码/直播”和“新闻综述”模板。

## 2026-07-11 · 版权与 ToS 路线：先做个人本地学习工具

- 背景：产品涉及下载音频转录（触碰 YouTube ToS）、保留原片片段插入讲解（对他人视频的再分发与演绎）、基于原视频内容重讲生成课程（独创性归属模糊）。这是产品对外化的最大合规风险，此前无任何决策。
- 决策：验证阶段按"个人本地学习工具"路线执行——本地运行、仅自用、不分发任何含原片素材的产物。对外产品化是独立决策，届时必须重新评估：助讲版能否只以"浏览器插件叠加层/时间戳跳转"形式实现而不复制原片；重讲版是否只用自产讲解素材。
- 理由：自用学习不阻塞验证；而合规路线会反向决定技术架构（本地 CLI vs 在线服务），不能在架构确定后再补。
- 待验证：对外路线下，官方字幕 API、创作者授权合作、纯跳转不复制三种方案的可行性。

## 2026-07-11 · 成功指标必须有测量协议

- 背景：初版成功指标全部是"用户能否……"的定性问句，无法判断黄金测试是否通过。
- 决策：每条指标配测量方式与通过标准，写入 `docs/需求与方案.md` 第 6 节；对照基线为"裸看同类视频"的耗时与复述质量。
- 理由：验证阶段用户是本人，主观感受极易自我说服；没有协议，测试结果不可信也不可比。
- 待验证：1/3 时间目标是否现实，首轮先以 1/2 为通过线。

## 2026-07-11 · 互动课程页是主形态

- 背景：文字、音频和视频各有优势，完整视频生成成本最高且不便于跳转追问。
- 决策：互动课程页作为学习母体；图文、音频和视频从同一课程结构派生。
- 理由：它最适合按需深入、跳过讲解、原文追溯和案例展开，也能复用中间产物控制成本。
