# 中文播放模式方案

> 建立日期：2026-07-13。对应 `docs/需求与方案.md` 的原片中文播放。当前状态：**10 分钟技术链已验证，翻译质量与学习效果待用户验证**。

## 1. 解决的问题

目标用户能看懂中文字幕，但持续盯字幕会遮挡代码、图表和产品演示，也无法在通勤或做简单操作时学习。中文播放要解决的是：**保留原视频的顺序、画面和证据，让用户主要靠听中文理解，并能随时回到原声核对。**

它不是逐字稿朗读，也不是英语学习功能。完成标准不是“每个英文词都翻译”，而是用户不看字幕也能跟上作者正在讲的概念，同时知道这仍是 AI 翻译音轨。

## 2. 三种播放模式的边界

| 模式 | 内容顺序 | 用户为什么用 | 是否替代原作者表达 |
|---|---|---|---|
| 原片中文 | 跟随原片 | 不盯字幕也能看完整原视频 | 只翻译，不补充解释 |
| 原片＋助讲 | 跟随原片，认知断点处插讲 | 原片讲得跳跃时补机制、背景和误区 | 原片与 AI 解释明确分轨 |
| AI 重讲课 | 按教学目标重组 | 用更短时间形成机制模型 | 使用自产脚本与画面，时间戳回原片 |

因此中文播放是原片模式下的一条音轨，不做独立课程入口。助讲和重讲继续承担“讲懂”，中文音轨只承担“听懂原片”。

## 3. 用户学习过程

### 3.1 入口与默认状态

课程播放器顶部提供模式：`原片` / `原片＋助讲` / `重讲课`。选择原片后，在播放器内选择音轨：

- `中文`：播放中文音轨，原视频静音；默认显示“AI 中文音轨”，中文字幕关闭；
- `原声`：播放原视频声音，中文字幕按用户上次设置显示；
- `原声＋中文低声`不提供，双声容易增加理解负担。

首次进入时若中文音轨已生成，默认使用中文；若未生成，保持原声并显示“生成中文音轨”，不伪装成可用状态。记住用户对音轨、字幕、语速的本地选择。

### 3.2 播放中的可观察行为

- 播放、暂停、拖动、章节跳转统一控制视频和中文音轨；页面只存在一个主播放状态。
- 中文音轨播放时，原视频保持静音，但画面、YouTube 控件和知识地图继续随原片时间运行。
- 用户点“核对原话”时，暂停中文，在同一时间点切回原声并展示该句中英文；关闭核对后回到中文音轨。
- 播放器持续标注 `AI 中文音轨 · 原片 00:18:42`，避免把译文误认为作者原声。
- 字幕是辅助层而非必需层：默认关闭中文；专名、代码和公式出现时可短暂显示“术语提示”，不把整句字幕重新铺回画面。

### 3.3 用户何时用它

标准学习路径为：导读判断价值 → 知识地图选范围 → 中文播放观看原片 → 难点切助讲或深讲卡 → 对观点有疑问时核对原声时间戳 → 课末闭卷复述。中文播放不取代课程结构，只降低进入原片的语言成本。

## 4. 音画同步策略

### 4.1 同步单位

不做逐字或口型同步，以 3—12 秒的“语义句”为最小单位。每句保存：

```json
{
  "source_start": 178.2,
  "source_end": 184.8,
  "source_text": "...",
  "zh_spoken": "……",
  "audio_file": "cue-0042.mp3",
  "audio_duration": 6.1,
  "terms": ["PRD"],
  "status": "reviewed"
}
```

一条中文句不能跨到下一个概念或下一次画面操作。时间戳来自原字幕，译文和音频都不能改写来源范围。

### 4.2 中文比英文更长时的优先级

1. 先把书面翻译改为自然、紧凑的口语，保留主张、条件、否定和数字；
2. 在允许范围内提高该句 TTS 语速，但不让全片忽快忽慢；
3. 使用原片中的停顿和无信息间隙；
4. 只在不影响代码操作和关键表情时，把原画局部降速到最低 0.9×；
5. 仍放不下则标记人工复核，不能删除限定条件或把解释提前到下一段。

允许 300—500ms 的句间误差；概念切换、代码操作、幻灯片翻页必须落在对应中文句内。首版不追求多人声纹复刻和口型匹配。

### 4.3 拖动与失步恢复

音频按语义句分片生成。用户拖动后定位到覆盖该时间点的句子，从句首播放；视频最多回退到该句 `source_start`，明确提示“已回到本句开头”。若音频和视频偏差超过 500ms，立即重定位当前音频片段；连续失败则自动降级到原声＋中文字幕。

## 5. 内容生成与质量门槛

生成链路：

```text
带时间戳英文字幕
→ 语义句切分
→ 术语表与说话人识别
→ 中文口语翻译
→ 时长预估与压缩
→ 分句 TTS
→ 音频时长回填与同步校验
→ 重点句人工抽查
```

中文口语翻译必须遵守：

- 框架名、API、模型名、论文名、代码符号保持原文，并有发音词典；
- 不添加作者没说的背景、评价或因果；必要的解释交给助讲；
- 保留不确定语气、反问、否定、数字、比较对象和观点归属；
- 广告和寒暄可按课程地图默认跳过，但必须在时间轴明确显示，用户可恢复完整播放；
- 多人访谈首版用两个明显可区分的中文声音；单人视频只用一个稳定声音，不克隆原作者声纹。

自动检查包括：字幕覆盖连续性、音频时长是否越界、数字和专名是否丢失、是否存在空音频、随机抽样回译。高风险句（数字、否定、代码步骤、争议观点）进入人工抽查队列。

## 6. 开源与产物边界

项目必须支持用户把仓库克隆到自己的电脑后，完成第三方视频的字幕获取、中文翻译、TTS 合成和本地播放。**第三方内容不随仓库分发，不等于项目不支持生成；恰恰相反，本地个人生成与播放是开源版的默认使用方式。**整个过程不依赖项目方服务器，用户的 API 密钥、字幕、译文和音轨都留在自己的电脑上。

目标本地工作流（待实现）：

```text
git clone 项目
→ 安装本地依赖并配置用户自己的模型/TTS 密钥
→ 输入 YouTube URL
→ 本地获取字幕与元数据
→ 本地生成中文口语稿和分句音轨
→ 启动本地网页，在原片画面上播放中文音轨
```

仓库开源生成管线、数据 schema、播放器和可自由使用的 fixture，不提交第三方视频、原音频、完整字幕或为特定视频生成的中文音轨。本地产物统一写入被 `.gitignore` 排除的工作目录；用户更新代码时可以保留缓存，清理缓存时也不会影响源码。默认命令不得自动执行 `git add`、上传、发布或遥测。

建议本地产物结构：

```text
local-data/<video-id>/
  source.json          # 标题、时长、字幕来源等元数据
  captions.en.json     # 带时间戳英文字幕
  captions.zh.json     # 中文短句字幕
  dubbing.json         # 语义句、术语、状态与同步信息
  audio/               # 分句中文音轨
```

公开课程优先支持三类来源：项目自有视频、明确授权视频、允许衍生与再分发的开放许可视频。普通 YouTube 链接只通过官方嵌入播放器展示；对其生成的中文音轨默认仅留在本地，不提供公共音轨 CDN。“个人技术分享”或“非商用”不是自动授权，也不能单独证明合理使用。YouTube 条款允许使用嵌入播放器，但对下载、修改或独立使用内容设有限制，因此此处采用保守产品边界；它不是法律结论。参考 [YouTube 服务条款](https://www.youtube.com/static?template=terms)、[版权与衍生作品说明](https://support.google.com/youtube/answer/2797466)、[合理使用说明](https://support.google.com/youtube/answer/9783148) 与 [CC BY 说明](https://support.google.com/youtube/answer/2797468)。

TTS 设计为适配器接口，但首版只提供一个默认实现和本地配置示例，不建立供应商市场。生成前显示预计字符数、音频时长、费用和磁盘占用；已有字幕、译文和音频均可断点复用。

## 7. 首版范围

首版只做：

- 单条已验证视频的原片中文音轨；
- 中文/原声切换、字幕开关、变速、拖动和章节跳转；
- 分句音频、术语词典、同步恢复与失败降级；
- “核对原话”切回同时间点；
- 用户克隆仓库后可用自己的配置完成本地生成与播放；产物进入忽略目录，不公开分发音轨。

首版不做：声纹克隆、口型同步、背景音乐/原声分离、逐词高亮、在线多人共享、实时边播边生成、全自动质量承诺。背景音乐或演示音效有教学价值的视频，首版保留原片画面但不混入原声；需要混音的场景留到授权素材验证后。

## 8. 真实视频验证

先用现有 70 分钟访谈样例截取连续 8—12 分钟完成端到端验证，再扩到四类视频。截取段应包含：两位说话人、一个专名密集段、一个观点限定句和一次画面/章节切换。

2026-07-13 技术验证：现有访谈样例 `02:58—12:58` 生成 79 条 3—12 秒 cue 和 79 个本地 MP3；滚动字幕重叠已消除，音频均落在 1.35× 同步上限内。桌面和 390px 下完成中文/原声切换、核对原话、时间线跳转、变速与同步恢复。由于本机没有 `OPENAI_API_KEY`，本轮中文稿使用项目已有缓存字幕作为验证种子并人工压缩 15 条越界句；真实 OpenAI 批量翻译质量仍未验证，不能据此声称内容质量通过。

| 目标 | 测量方式 | 通过标准 |
|---|---|---|
| 不看字幕也能理解 | 关字幕只听中文，结束后闭卷复述主张、依据和限制 | 三项均可被原片印证，无关键反转 |
| 音画同步 | 抽查 10 个概念/画面切换点 | 10 处均未跨到下一概念；关键操作误差 ≤ 500ms |
| 可追溯 | 随机点 5 次“核对原话” | 5 次都在对应语义句开头，能看到中英文依据 |
| 术语准确 | 抽查所有专名、数字、否定与代码符号 | 无错误替换或遗漏 |
| 听感可持续 | 连续听完测试段，记录手动切原声、倒退和看字幕次数 | 非核对目的的被迫切换合计 ≤ 2 次 |
| 生成成本 | 记录字幕字符、模型 Token、TTS 字符、耗时和磁盘 | 能显示真实数据；首轮不设拍脑袋价格线 |

通过访谈片段后依次验证工程教程、模型解读和产品评测。工程教程若无法在不看屏幕时理解不算失败，但中文句必须准确对齐代码操作；该类型的目标是“看画面、无需读字幕”。

## 9. 待验证问题

- 用户更喜欢全片一个稳定中文声线，还是访谈按说话人分声线；
- 0.9× 局部画面降速是否比稍快的中文语速更自然；
- “核对原话”应按住临时切换，还是点击进入核对状态；
- 广告、寒暄默认跳过是否破坏用户对原片完整性的信任；
- 原片中的关键音效和情绪信息，何时值得在有授权前提下做原声压低混音。
