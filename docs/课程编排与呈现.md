# 课程编排与呈现

> 本文回答三个核心问题：哪些内容值得讲、怎样控制长视频成本、不同内容应该以什么课程形态呈现。
>
> 每节标注验证状态：**纯推演**（未接触真实视频）、**待验证**（有测试计划）、**已验证**（有真实样例支持）。验证一节改一节；黄金测试完成前不再新增设计。

## 1. 不是每个章节都要讲解

状态：**纯推演**。五种动作和评分维度均未在真实视频上标注过。

系统先还原视频结构，再决定每一段的处理动作。章节只是内容边界，不等于讲解单元。每个语义段落被标记为以下一种：

| 动作 | 适用内容 | 默认呈现 |
|---|---|---|
| 深讲 | 核心机制、关键论证、理解门槛高且影响后文 | 2—6 分钟讲解 + 图示/案例 |
| 快讲 | 重要但容易理解的观点或背景 | 30—90 秒概括 |
| 原片 | 演示、个人故事、作者表达本身有价值 | 保留原片 + 字幕，不插讲 |
| 摘要 | 重复论证、次要背景、长铺垫 | 一两句话，可展开 |
| 跳过 | 广告、寒暄、重复、离题、低信息问答 | 时间轴保留，默认不播放 |

### 1.1 讲解价值评分

每段按五项判断，而不是只看关键词：

- **重要性**：是否支撑视频主结论；
- **理解难度**：目标用户是否容易误解或缺少前置知识；
- **新颖度**：是否超出常见知识或提供作者独特判断；
- **可迁移性**：能否影响用户自己的项目决策；
- **证据密度**：是否包含机制、数据、演示或能验证观点的案例。

高重要性 + 高难度的内容进入深讲；高重要性 + 低难度进入快讲；低重要性内容默认摘要或跳过。新颖但有争议的观点要保留作者语境，不能被 AI 改写成事实。

### 1.2 用户控制

系统给推荐，但用户始终可以：

- 一键选择“只看核心”“标准课程”“完整精读”；
- 对任意模块选择跳过讲解、只看原片、稍后学习；
- 在播放中关闭后续助讲；
- 调整单个模块为“讲简单点”或“讲深入点”；
- 预览本次会插入多少讲解、预计新增多长时间。

默认不应在每章后强制插话。助讲是一层可开关的课程轨道。

## 2. 长视频的 Token 与成本控制

状态：**待验证**。2026-07-17 已实现课程/中文翻译的内容指纹缓存和成功/失败 usage 账本，但仍没有 3—5 条真实视频的 Token 与成本样本；分层模型方案继续视为推演，不据此定价。

一小时英文口语通常只是数千到一万余词，完整文字可以被现代长上下文模型处理，但如果每次提问都把整份逐字稿重复发送，成本会迅速放大。关键不是“能不能塞进去”，而是只处理一次、后续按需读取。

### 2.1 分层处理

```text
原字幕/转录
→ 语义切片与术语表（一次）
→ 每片段事实卡（一次、小模型）
→ 全局课程地图（只读事实卡）
→ 选择少量深讲模块（强模型）
→ 用户提问时只检索相关片段
```

保存的中间产物包括：带时间戳原文、片段摘要、实体与术语、观点归属、案例、章节关系和检索索引。更换讲解语言或深度时优先复用这些产物，不重新理解整段视频。

### 2.2 模型分工

- 字幕清洗、切片、去重、简单分类：便宜快速的模型；
- 全局结构和重点判断：能力较强的长上下文模型，只运行一至两次；
- 深度讲解、案例生成、争议判断：只对选中的核心模块使用强模型；
- 视频问答：检索相关原文与相邻上下文，不重复提交全文；
- 语音和画面生成：从已确认的课程脚本生成，不让视频模型重新理解原视频。

### 2.3 成本对用户可见

处理前展示视频时长、预计模块数、预计深讲数量和课程时长。产品提供三档：快速导读、标准课程、完整精读。具体 Token 和费用不能现在拍脑袋承诺，必须用真实视频记录字幕长度、各阶段输入输出 Token、缓存命中率和重跑次数后再定价。

## 3. 课程不是只有文字稿

状态：**纯推演**。MVP 只做互动课程页的手工等价物（结构化文档）；图文、音频、视频三种形态在学习闭环验证前不投入。

同一份课程结构可以生成四种互相同步的学习界面。

### 3.1 互动课程页（默认主形态）

播放器、知识地图、讲解卡和时间戳逐字稿同时存在。模块可以折叠，观点可追问，案例可展开，点击原文回到视频。这是信息最完整、最适合深度学习的形态。

### 3.2 图文课程

不是逐字稿，而是经过教学重组的短章节：问题 → 观点 → 机制图 → 案例 → 工程启示 → 最小实践。适合快速阅读和复习，也作为后续音频、视频的内容母版。

### 3.3 音频课程

基于课程脚本生成 10—30 分钟中文讲解，可在通勤时听。需要口头描述图示和代码，适合观点、访谈和模型解读，不适合依赖界面演示的工具教程。

### 3.4 视频课程

视频不是让数字人朗读摘要，而是把课程脚本视觉化。基本镜头单位是“一个教学意图”，常用画面包括：

- 原视频关键片段：保留作者语气、演示和证据；
- 讲师旁白：负责衔接、解释和指出价值；
- 机制动画：逐步展示流程、架构、数据流和因果关系；
- 代码聚焦：只显示当前讲到的关键行，配执行结果或前后对比；
- 产品演示：裁切界面并标注关键操作；
- 对比卡片：旧方法/新方法、不同模型、不同观点；
- 案例白板：背景、输入、过程、反馈和结果逐步出现；
- 章节检查：用一道判断题或一句复述提示帮助停顿。

## 4. 两种视频编排方式

状态：**纯推演**，且整节在 MVP 之外。原视频助讲版还受版权路线约束（见决策日志"版权与 ToS 路线"）。

### 4.1 原视频助讲版

目标是保留优秀创作者本人的表达，只在认知断点处介入。

```text
10—30 秒课程导览
→ 原片关键段落
→ 必要时冻结画面
→ 30—90 秒中文解释/机制图/案例
→ “接下来注意作者如何处理 X”
→ 回到原片
```

介入触发条件：出现关键术语但未解释、论证跳步、引用隐含背景、从概念突然进入实现、案例与结论关系不清。寒暄、重复和广告直接跳过；清楚的段落不插讲。用户可以关闭全部助讲，只保留章节导航和字幕。

### 4.2 AI 教师重讲版

目标不是按原视频顺序缩写，而是像优秀课程或技术博主一样重新建立叙事：先制造问题，再给模型，随后用案例验证，最后落到项目决策。

```text
钩子：你会在哪个真实项目中遇到这个问题
→ 学习承诺：学完能判断或完成什么
→ 必要背景：只补后文真正需要的
→ 核心模型：一张机制图讲清
→ 案例：完整跑一遍输入、过程、反馈、结果
→ 回到作者：其独特观点和证据
→ 取舍与边界：什么时候不要使用
→ 最小实践：用户下一步可以做什么
→ 三句话复盘
```

大学课程的优势是结构、概念递进和严谨；优秀技术博主的优势是问题意识、节奏、视觉化和案例。产品应组合两者，而不是模仿课堂时长或博主人设。

## 5. 不同视频使用不同课程模板

状态：**待验证**。四个模板对应黄金测试的四类视频，逐条测试后修订或合并。

分类必须现在定义，否则同一提示词会把不同视频都处理成相似摘要；但不用现在穷举所有类型。首版建立四个主模板，允许混合视频按章节选择不同模板。

### 5.1 AI 技术/工程教程

重点：问题、系统结构、关键代码、执行过程、错误与取舍。呈现以代码聚焦、流程动画、输入输出对比和最小复现为主。需要区分“作者演示成功”和“方案在真实项目中可靠”。

### 5.2 模型/论文/理论解读

重点：它改变了什么、机制如何运作、证据是否支持、与上一代有什么差异、能力边界。呈现以机制图、对比表、实验结果和反例为主。避免照读参数与 benchmark。

### 5.3 AI 新产品/工具评测

重点：为谁解决什么任务、实际操作链路、关键能力、体验证据、限制、成本和替代品。呈现以产品演示、任务前后对比和真实场景为主。避免把发布宣传当成实测结论。

### 5.4 访谈/圆桌/观点视频

重点：说话人主张、论据、分歧、隐含前提和可迁移判断。先按观点而非时间顺序重组，再保留关键原话。呈现以人物观点卡、论证链、观点对比和事实核查为主。不要强行把每段聊天变成知识点。

### 5.5 混合内容

先判断全片主类型，再给章节标子类型。例如一段模型访谈可能包含理论解释、产品演示和个人观点；课程地图分别套用对应模块，而不是整片只用一个模板。

## 6. 首轮黄金测试

状态：**进行前提**。这是本文所有"纯推演"内容转为"已验证"的唯一途径，优先级高于任何新设计。首条测试按 `docs/需求与方案.md` 的测量协议执行。

至少选择四条真实视频，每种模板一条。每条先人工确认：哪些段落应深讲、快讲、保留原片、摘要和跳过；再与系统结果比较。评估重点不是摘要是否流畅，而是内容选择是否正确、课程编排是否帮助理解、额外时长是否值得。
