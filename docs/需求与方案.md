# 需求与方案

> 建立日期：2026-07-11。本文记录产品为什么存在、为谁服务以及如何解决问题。

## 1. 需求意图

用户每天同时面对两类信息负担：一类是 AI 新闻、产品发布、模型更新、X 帖、邮件订阅、文章、播客和视频，来源分散且无法逐条完整消费；另一类是 YouTube 上 20 分钟到 2 小时的 AI Engineering、AI Builder 和技术创作者长视频，技术密度高，缺少适合个人水平的课程结构。当前往往先用 Gemini 或 NotebookLM 获取概要，再回到原文或视频寻找重点；遇到难点还要反复暂停并另问 AI。

产品目标分成连续两步：先把用户投递的多源 AI 内容转换成可连续收听、同步扫读、逐句追溯的短资讯，帮助用户决定跳过、收藏、查证或深入；再把值得投入的长视频转换成可浏览、可跳转、可追问、可看案例、可实践、可重新讲解的进阶课程。

## 2. 目标用户

默认用户画像：

- 做过基础开发项目，能阅读代码和使用常见开发工具；
- 主要依靠 AI 完成 vibe coding，能够做出东西，但知识体系不完整；
- 希望进阶理解架构、原理、工程权衡和技术边界；
- 每天接触大量 AI 内容，希望先快速筛选，再把有限时间投入真正重要的内容；
- 不想从零基础教材重新学，也无法直接吸收专家之间高度压缩的讨论。

非目标：纯英语学习、泛娱乐资讯或视频总结、完全零基础编程教学、只提供逐字稿翻译、自动聚合全网新闻、用摘要替代事实核查。

## 3. 核心价值

一句话定位：先把多源 AI 内容压缩成可追溯的个人资讯电台，再把值得投入的长视频重构成适合 vibe coding 开发者进阶的结构化课程。

产品解决四层问题：

1. **值不值得继续看**：用 20—60 秒知道一条资讯发生了什么、新意、价值、限定和来源。
2. **值不值得学**：快速知道长视频主题、作者主张、含金量、适合程度和重点片段。
3. **具体讲了什么**：按语义生成知识地图，分离概念、观点、实现、案例与争议。
4. **如何真正学懂**：提供通俗解释、技术机制、案例、实践任务和带时间戳的追问。

## 4. 标准信息获取与学习流程

### 4.0 多源资讯速听

用户可以批量粘贴文字、加入公开链接，或导入邮件、图片、PDF、音视频文件。系统提取正文、OCR、字幕或转录，保留文章段落、PDF 页码、图片区域、邮件位置或音视频时间戳；提取不完整时必须显式标记，不得假装得到全文。

重复报道按同一事件聚合。每条内容生成 20—60 秒、复杂项最多 90 秒的单人短讲，先让初听者建立最小上下文，再帮助判断价值。默认讲述顺序是：主体与背景、内容与主张、原理或趋势、判断与限定；不同内容类型可以调整顺序，但必须让未读原文的用户回答“谁在讲、这是什么、传递了什么”。事实、作者观点、AI 解释和 AI 判断必须分开标注，文字与语音共用同一份脚本。用户可以连续播放多条内容，随时跳过、收藏、打开原文、查证或讲深一点；感兴趣的长视频进入后续课程流程。

### 4.1 导入与校正

输入 YouTube 链接后，优先获取官方或作者字幕；没有字幕时再下载音频转录。保留带时间戳的英文原文，生成自然中文，并二次校正框架名、API、模型名、论文名和代码术语。

### 4.2 课程导读

先输出一张紧凑的课程决策卡：一句话主题、作者的 3—5 个核心主张、新信息、适合人群、前置知识、最值得看的片段、可跳过部分和预期收获。

### 4.3 知识地图

根据语义拆成 5—12 个主题，不按固定时长切片。每个主题标记时间范围、重要度、难度、内容类型和推荐学习方式。

### 4.4 模块化学习

每个模块统一回答：

1. 这部分解决什么问题；
2. 作者的核心判断；
3. 用人话怎么理解；
4. 背后的技术机制；
5. 原视频用了什么案例；
6. 还需要什么补充案例；
7. 对 vibe coding 开发者有什么用；
8. 如何在自己的项目中做最小实践；
9. 对应原文和时间戳。

解释提供三个深度：30 秒理解、通俗讲解、技术深入。默认从通俗讲解开始，但不回避必要术语。

### 4.5 案例与实践

案例分为视频原案例、AI 补充案例和失败反例。案例必须说明它在证明什么机制，并包含背景、输入、关键步骤、判断或反馈方式、结果和适用边界。每个重要模块尽量提供一个 15—30 分钟可完成的最小实践。

### 4.6 可追溯问答

用户可以针对整期视频或某个主题提问。回答必须区分：作者原话、AI 归纳、AI 解释、外部补充；涉及原视频的结论附时间戳。

### 4.7 三种中文播放与讲解模式

- **原片中文播放**：保留原视频顺序和画面，用与语义句时间轴同步的中文音轨替代原声；用户可随时切回同一时间点核对原话。它只做翻译，不补充作者未说的解释。
- **原视频助讲模式**：保留原视频，每个知识段落结束后插入 30—90 秒中文助讲，解释本段内容、价值、前置知识和常见误区。
- **AI 教师重讲模式**：重新组织整期内容，生成 10—30 分钟课程，结构为导入、主题讲解、机制图、案例、实践提示和总结。

三种模式共享课程页、知识地图和时间戳追溯。默认优先让用户听中文；原声与中文字幕由用户主动切换。详细设计见 `docs/中文播放模式方案.md`。

## 5. MVP

产品当前有两个相互独立的验证 MVP：资讯速听先验证批量判断，课程 MVP 继续验证长视频理解。任一层通过都不能替另一层证明成功。

### 5.1 课程 MVP

课程第一版只验证一件事：用户能否用原来约三分之一的时间，判断视频价值并理解关键机制。

MVP 只包含四项：课程导读、知识地图、2—3 个深讲模块（含原案例）、模块到原视频的时间戳追溯。链接导入允许手工完成（人工获取字幕），不要求自动化。

验证通过后再议：双语字幕自动化、补充案例、时间戳问答、最小实践生成、完整中文音轨。

明确不做：数字人、复杂视频剪辑、社区、课程市场、多人协作、完整在线编程环境。

### 5.2 资讯速听 MVP

资讯速听第一版只验证一件事：用户能否用裸读或裸看四分之一以内的时间，可靠判断一批内容中哪些值得继续投入。

首版只包含：手工批量投递文字、链接和本地文件；网页提取、OCR、字幕或转录；带证据锚点的短资讯卡；单人短语音与同步文字；连续播放以及跳过、收藏、原文、深挖动作。邮件用复制正文或 `.eml` 文件导入。

手工真实邮件验证通过后，允许增加一个严格受限的 Gmail 自动入口：只在用户自己的 Mac 上按发件人白名单和时间窗口读取，使用用户配置的第三方模型密钥提炼，并把私人产物同步到带 HTTPS 和认证的个人服务器。密钥、邮件正文、摘要和音频不得进入公开 Git 或公开 Pages。

仍明确不做：RSS、X 账号自动同步，自动关注全网新闻，黑箱个性化推荐，双主播播客，默认外部搜索，完整知识库或复杂 RAG。新的自动连接器仍需在现有真实样例验证通过后单独决策。

## 6. 成功指标与测量协议

验证阶段用户就是本人，每条指标必须有可执行的测量方式，不接受"感觉不错"：

| 指标 | 测量方式 | 通过标准 |
|---|---|---|
| 3 分钟决策 | 只看课程导读，计时后写下"学/不学 + 理由"，再看完原片核对 | 决策与看完原片后的判断一致 |
| 主题划分与术语准确 | 对照原片人工标注每个主题的边界和术语 | 无错误归属；术语拼写与指代全部正确 |
| 理解关键机制 | 学完课程后合上材料，用自己的话写核心机制复述 | 复述能被原片对应片段印证，无关键性错误 |
| 可追溯 | 抽查 5 处结论，点回时间戳核对 | 5 处全部能在 ±30 秒内找到原始依据 |
| 时间下降 | 记录课程总学习时长，与原片时长对比 | 总时长 ≤ 原片的 1/2（目标 1/3） |

对照基线：先裸看一条视频并记录理解耗时与复述质量，再用手工课程学同类另一条，两者对比。案例质量和最小实践的指标在 MVP 验证通过后再定义。

资讯速听单独使用以下协议，不用“听过摘要”冒充机制学习：

| 指标 | 测量方式 | 通过标准 |
|---|---|---|
| 批量判断速度 | 对同一批 10 条内容分别记录裸读/裸看与速听判断时间 | 速听 ≤8 分钟且总时间 ≤基线的 1/4 |
| 兴趣判断一致 | 速听后选择打开/收藏/跳过，完整消费后再次选择 | 一致率 ≥80%；最终认定的“必看”内容 0 漏判 |
| 保真与归属 | 每条抽查 2 个事实或判断并返回来源核对 | 数字、否定和条件无反转；作者观点与 AI 判断 0 混淆 |
| 批次理解 | 结束后闭卷复述最重要 5 条的“发生了什么＋一个观点” | 至少 4/5 可被来源支持 |
| 连续听感 | 连续收听 8—10 分钟，记录听不清导致的回退或切文字 | 合计 ≤2 次；专有名词误读 0 次 |
| 深挖路径 | 从感兴趣内容进入依据或后续学习 | 2 次操作内到原文锚点；长视频可进入课程导读 |

## 7. 待验证问题

- 不同类型视频需要几套课程模板；
- 多长的助讲不会破坏原视频节奏；
- 中文解释应该补充多少视频外背景；
- 代码画面、幻灯片和口述内容如何联合解析；
- 讲解音频与完整讲解视频的价值差异是否足以覆盖生成成本。
- 资讯速听的 20—60 秒是否足以保留不同内容类型的关键限定；
- 同一事件跨新闻、X、邮件和音视频的聚类是否会误合并不同观点；
- 手工投递验证通过后，RSS、邮件转发和账号连接器的优先顺序；
- 用户的跳过、收藏和打开行为是否足以支持可解释排序，而不形成过滤泡。
