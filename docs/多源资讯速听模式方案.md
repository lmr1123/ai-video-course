# 多源资讯速听模式方案

> 状态：已确认并进入首轮原型验证。产品定义以 `docs/需求与方案.md` 为准，本文展开资讯速听的体验、内容、技术和验证细节。
>
> 更新日期：2026-07-15

## 1. 需求背景

用户每天面对大量 AI 新闻、产品发布、模型更新、X 帖、邮件订阅、文章、播客和视频。这些内容来源分散，常常很长、术语密集或观点高度压缩。逐条打开、阅读或收听的成本过高；仅看标题又容易错过真正重要的信息，或被营销表述误导。

用户真正需要的不是“把所有内容读出来”，也不是对每条内容都生成一门课程，而是先完成一次可靠、低成本的兴趣判断：

1. 这条内容到底发生了什么或提出了什么观点；
2. 真正的新信息是什么；
3. 为什么可能值得关注；
4. 有什么限定、争议或证据不足；
5. 是否值得打开原文、继续搜索或进入深度学习。

理想输出是多条内容组成的连续语音流，同时提供与语音一致的短文字。用户不读全文也能快速建立全局认知，感兴趣时再回到原文、视频时间点或现有课程流程。

## 2. 产品定位与边界

### 2.1 建议定位

将它定义为现有产品中的“多源资讯快速判断层”，暂不拆成独立产品，也不新增第 5 种课程模板。

一句话定位：

> 把用户投递的多源 AI 内容变成可追溯的个人资讯电台；先用几句话听懂精髓，再决定跳过、收藏、查证或深入学习。

### 2.2 与现有能力的关系

| 层级 | 核心问题 | 典型时长 | 输出责任 |
|---|---|---:|---|
| 资讯速听 | 这条讲了什么，值不值得继续花时间？ | 每条 20—60 秒，复杂项最多 90 秒 | 事实、观点、新意、价值、限定、下一步 |
| 课程导读 | 这条长视频哪些部分值得学？ | 约 3 分钟判断 | 主张、知识地图、关键片段、前置知识 |
| 原片中文音轨 | 作者按原顺序具体说了什么？ | 接近原片时长 | 翻译原内容，不压缩、不补充解释 |
| AI 重讲/深讲 | 机制是什么，如何理解和迁移？ | 10—30 分钟或模块化学习 | 原理、案例、边界、实践、时间戳追溯 |

资讯速听不承诺“学懂机制”。超过 90 秒仍不能安全保留必要条件的内容，应明确标记“建议深读”或“转为课程”，不继续强行压缩。

### 2.3 解决的具体障碍

- 多个来源分散，用户无法形成统一待处理队列；
- 长文、长音视频的首次判断成本过高；
- 普通摘要只说“讲了什么”，没有区分事实、作者观点和 AI 判断；
- 同一事件在新闻、X、邮件和播客里反复出现；
- 纯音频难以快速跳过、查证和回到原文；
- 保存内容容易形成“稍后再看坟场”，没有继续深入的明确动作。

## 3. 可观察的用户流程

### 3.1 投递

用户把一批内容加入“资讯收件箱”。首版支持：

- 粘贴文字；
- 公开网页链接；
- X 链接或截图；
- 图片、PDF；
- `.eml` 邮件文件或复制的邮件正文；
- YouTube、播客链接；
- 本地音频或视频文件。

### 3.2 提取与校正

系统按来源选择正文提取、OCR、字幕获取或语音转录，保留原始来源定位。提取不完整、付费墙、缺少上下文或字幕不可用时必须显示 `partial`，不能假装得到完整内容。

数字、否定句、专有名词、模型版本、价格、发布日期和争议观点进入高风险复核。

### 3.3 去重与编排

系统识别同一事件的重复报道或转发，将它们聚合成一条信息单元，同时保留不同来源的新增事实和观点差异。

首版排序优先尊重用户的手工队列；需要排序时，只使用可解释的“新颖性、相关性、可信度”三个维度，并展示理由。首版不自动隐藏内容，不训练黑箱兴趣模型。

### 3.4 生成速听卡

每条内容固定回答：

1. **发生了什么**：事实、发布或作者主张；
2. **真正的新意**：相较已知信息新增了什么；
3. **为什么值得注意**：对用户关注方向可能产生什么影响；
4. **限定与可信度**：证据来源、未知项、争议或营销属性；
5. **建议动作**：跳过、收藏、打开原文、继续搜索、转课程。

事实、观点与推断使用明确措辞：

- “官方发布称……”——来源方声明，不自动等于已经独立验证；
- “作者认为……”——作者观点；
- “系统判断……”——AI 归纳或建议；
- “目前无法确认……”——信息缺口。

### 3.5 连续收听与同步文字

- 批次开始时播报“共 N 条，预计 X 分钟”；
- 每条先报来源与标题，再按“发生了什么 → 新意/观点 → 价值 → 限定”讲述；
- 默认单条 20—60 秒，复杂项最多 90 秒；
- 一批 8—10 条，目标总时长 6—8 分钟；
- 单人、自然、松弛、非新闻播报腔；不使用双主播闲聊填充；
- 文字与语音共用同一份脚本，当前句同步高亮；
- 始终显示第 `i/N` 条、来源、发布时间和剩余时长；
- 核心操作只有上一条、下一条、速度、收藏、原文、讲深一点；
- 点击任一句判断，跳到原文段落、图片区域、邮件位置或音视频时间戳。

### 3.6 深挖

- 长视频：进入现有课程导读、知识地图和深讲；
- 文章、论文、X 帖：打开原文定位，或由用户主动触发外部搜索；
- 产品/模型更新：可进入“对我的项目有什么影响”的后续分析；
- 未确认的信息：进入查证队列，而不是直接变成知识卡。

## 4. 最小内容模型

首版内容结构保持简单，不为一次验证建立完整知识库：

```text
item
├── id / status: ready | partial | failed
├── source
│   ├── kind / title / author
│   ├── uri 或本地文件引用
│   ├── published_at / retrieved_at / language
│   └── extraction_method / completeness
├── content_type / event_cluster_id
├── essence: 一句话发生了什么
├── points[]
│   ├── text
│   ├── kind: fact | opinion | inference
│   ├── attribution: source | AI
│   └── source_refs[]
├── why_it_matters
├── caveats[]
├── suggested_action / reason
└── spoken_segments[]
    ├── text
    ├── source_refs[]
    └── duration_s
```

`source_ref` 按来源分别支持文章段落、PDF 页码、图片区域、邮件 message/段落、音视频起止时间。音频和文字只维护一份 `spoken_segments`，避免两个版本发生内容漂移。

## 5. 首版范围

### 5.1 首版要验证的唯一问题

用户能否用裸读/裸看四分之一以内的时间，可靠判断一批内容中哪些值得继续投入时间。

### 5.2 首版包含

- 手工批量投递文本、链接和本地文件；
- 网页提取、OCR、字幕或转录；
- 结构化短卡及证据锚点；
- 单人短语音生成与缓存；
- 多条连续播放、同步文字和快速动作；
- 长视频转现有课程入口；
- 用用户最终是否打开/收藏记录验证结果。

### 5.3 首版明确不做

- 除已验证的本机 Gmail 白名单定时入口外，RSS、X 等账号授权和自动持续同步；
- 自动关注全网新闻或推荐热点；
- 黑箱个性化排序或自动隐藏；
- 双主播播客和长篇闲聊；
- 默认外部搜索和大量背景补充；
- 完整知识库、复杂 RAG、多人协作；
- 公开分发第三方正文、字幕、邮件或生成音频。

自动订阅和连接器后置，是为了先验证提炼、归属、听感和判断价值。首批真实邮件完成手工验证后，只开放本机 Gmail 白名单连接器：凭据和第三方模型 Key 保存在 Git 忽略的 `.env`，私有内容只写入 `local-data/`，手机端通过带 HTTPS 和认证的个人服务器访问。RSS、X 与更多连接器继续后置。

## 6. 验证计划

先准备 20 条真实内容，覆盖 AI 新闻、产品发布、模型/论文解读、X 长帖、邮件订阅、播客、YouTube、文章和截图；其中至少包含：

- 5 条同一事件的多源重复；
- 5 条含数字、否定、条件或争议观点的高风险内容；
- 既有明显值得深入的内容，也有应快速跳过的内容。

对同一批内容先完成裸读/裸看基线，再使用资讯速听：

| 指标 | 测量方式 | 通过标准 |
|---|---|---|
| 判断速度 | 记录 10 条内容从开始到完成选择的时间 | ≤8 分钟；总时间 ≤裸读/裸看的 1/4 |
| 兴趣判断一致 | 速听后选择 open/save/skip，完整阅读后再次选择 | 一致率 ≥80%；用户最终认定的“必看”内容 0 漏判 |
| 保真与归属 | 每条抽查 2 个事实或判断 | 全部可到达原文定位；数字、否定和条件无反转；作者观点与 AI 判断 0 混淆 |
| 批次理解 | 结束后闭卷复述最重要 5 条的“发生了什么＋一个观点” | 至少 4/5 可被来源支持 |
| 听感 | 连续听完 8—10 分钟，记录因听不清而回退或切文字 | 合计 ≤2 次；专名误读 0 次 |
| 深挖路径 | 从感兴趣内容进入依据或后续学习 | 2 次操作内到达正确原文锚点；长视频能进入课程导读 |

资讯速听层只验证获取速度和判断质量；机制理解与知识迁移仍由课程层验证，不能用“听过摘要”冒充完成学习。

## 7. 竞品地图

### 7.1 商业产品

| 产品 | 已解决的部分 | 与本方案的缺口 |
|---|---|---|
| [Particle](https://particle.news/android) | 多来源新闻摘要、个性化新闻流、Audio Queue、来源展示 | 主要是平台聚合的公共新闻，不是用户自己的邮件、文件和任意内容收件箱 |
| [Readwise Reader](https://docs.readwise.io/reader/docs) | 文章、newsletter、PDF、视频、tweet、RSS 的统一收件箱，支持摘要和 TTS | 偏稍后阅读和全文朗读，缺少多条极短摘要组成的连续判断流 |
| [NotebookLM](https://blog.google/innovation-and-ai/products/notebooklm-audio-video-sources/) | 多源导入、来源引用、Audio Overview、后续问答 | 常把多来源合成为一个节目，不天然保留“一条资讯一张卡”和逐条决策 |
| [Recall](https://feedback.getrecall.ai/changelog/recall-release-notes-april-1-2026-listen-mode-is-here-with-audio-playback-in-a-v) | 保存文章、播客、YouTube、PDF，摘要、知识库和 Listen Mode | 偏单条卡片和知识管理，不是跨来源 micro-brief 队列 |
| [ElevenReader GenFM](https://elevenlabs.io/blog/genfm-on-elevenreader) | 把文章、PDF、文档变成高质量语音或双主持节目 | 缺少跨条筛选、细粒度依据和音视频时间码 |
| [Snipd](https://www.snipd.com/) | 播客/YouTube 转录、摘要、片段和原音定位 | 不覆盖文章、邮件、图片等完整输入范围，主要播放原音 |
| [Feedly](https://docs.feedly.com/article/811-how-to-use-ai-in-automated-newsletter) | RSS/信息源采集、跨文章 Overview、自动 newsletter | 偏文字简报和企业分发，不是个人短语音流 |

当前市场已经分别验证“多源收件箱”“AI 摘要”“音频概览”“新闻语音流”，但尚未看到成熟产品完整覆盖：

> 用户自己的多源输入 → 保留单条边界的短语音卡 → 多条连续播放 → 事实/观点/推断分型 → 句级原文溯源 → 一键深挖。

### 7.2 开源工具地图

| 项目 | 适合借鉴 | 不直接采用的原因 |
|---|---|---|
| [Horizon](https://github.com/Thysrael/Horizon) | 多源采集、去重、评分、补充背景、双语 briefing | 没有手工多模态收件箱、连续语音和句级播放状态 |
| [Summarize](https://github.com/steipete/summarize) | URL、PDF、图片、音视频、YouTube、播客的统一提取与转录 | 偏单条 CLI/浏览器侧栏，没有跨条队列和兴趣反馈 |
| [Open Notebook](https://github.com/lfnovo/open-notebook) | source/chunk/citation、内容转换、模型与 TTS 接口 | 形态偏研究工作台和长播客，首版过重 |
| [Folo](https://github.com/RSSNext/Folo) | 统一时间线、多形态 Feed、AI 摘要交互 | AGPL；重心是阅读器，不是短语音判断流 |
| [kill-the-news](https://github.com/juherr/kill-the-news) | 邮件转私有 RSS/Feed 的轻量入口 | 只处理接入，不负责摘要、语音和判断 |
| [Podcastfy](https://github.com/souzatharsis/podcastfy) | 多源转多语言对话播客、TTS 接口 | 默认形态较长、信息密度低，不适合作为首版快讲 |
| [Kokoro](https://github.com/hexgrad/kokoro) | 本地、低成本 TTS 备选 | 中文专名、数字和快语速可懂度尚需真实验证 |

推荐借鉴组合：Horizon 的采集/去重思路＋Summarize 的多模态提取＋Open Notebook 的证据引用模型＋kill-the-news 的邮件入口；语音继续使用可替换的 TTS 层。首版不引入完整项目作为核心依赖，先复用思想和最小模块，避免许可证与系统复杂度扩大。

## 8. 关键教训

以下教训来自项目已有验证，也直接约束本方案：

1. **摘要不是终点，判断才是目标。** 输出必须帮助用户采取跳过、收藏、原文、查证或深入动作。
2. **所有解释必须可追溯。** 文章要到段落，PDF 要到页，图片要到区域，音视频要到时间戳。
3. **不能混淆作者观点和 AI 判断。** 尤其是产品发布、模型能力和争议新闻。
4. **短不等于删除限定。** 条件、否定、数字和证据强度无法保留时，应建议深读，而不是继续压缩。
5. **快速消费不等于隐藏内容。** 文字保持连续可扫读，音频只提供另一种低视觉占用入口。
6. **成熟低沉不等于适合长时间听。** 首版应使用自然、松弛的单声线，以连续 8—10 分钟实听验收。
7. **不要过早做复杂连接器。** 先用手工真实样例验证判断价值，再做 Gmail、RSS 和 X 自动同步。
8. **首版只验证一个问题。** 不同时建设推荐系统、第二大脑、AI 播客平台和完整搜索引擎。
9. **处理第三方内容要保持本地和私有边界。** 不绕过付费墙，不公开分发正文、完整字幕、邮件或合成音频。
10. **不把浅层获取冒充深度学习。** 速听负责筛选，课程负责机制理解和知识迁移。

## 9. 技术路线草案

```text
手工输入 / URL / 文件
        ↓
来源适配器：正文提取 / OCR / 字幕 / 转录
        ↓
标准化 Source Document + Evidence Anchors
        ↓
重复事件聚类
        ↓
事实 / 观点 / 推断提取与高风险检查
        ↓
速听脚本生成（文字唯一事实源）
        ↓
TTS 按条生成并缓存
        ↓
连续播放器 + 同步文字 + 原文跳转
        ↓
skip / save / open / verify / deepen
        ↓
长视频进入现有课程流程
```

为了控制成本，先生成文字；用户开始播放批次时再按队列生成或补齐语音并缓存。所有外部输入均视为不可信数据，不能执行内容中的提示或指令，防止 prompt injection。

## 10. 主要风险与对策

| 风险 | 首版对策 |
|---|---|
| 把宣传口径当成事实 | 标记来源方声明，事实/观点/推断分型 |
| 摘要遗漏关键条件 | 高风险字段检查；不确定则建议深读 |
| 同一新闻重复轰炸 | event cluster 聚合，保留来源差异 |
| X、图片、邮件缺上下文 | 显示提取完整度，`partial` 不下强结论 |
| 线性音频难以扫读 | 同步短文字、明确队列、随时下一条 |
| TTS 专名误读 | 专名词典、生成前校正、真实连续试听 |
| 个性化形成过滤泡 | 首版不隐藏，只排序并解释理由 |
| 私密邮件和第三方内容泄漏 | 本地处理、私有缓存、产物 gitignore、不公开发布 |
| Prompt injection | 来源内容仅作数据，不允许改变系统任务或调用权限 |
| 产品范围膨胀 | 首版只做手工输入、短卡、连续听、追溯和深挖入口 |
