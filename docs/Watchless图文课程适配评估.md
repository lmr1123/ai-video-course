# Watchless 图文课程适配评估

> 评估日期：2026-07-19。目标仓库：[`chenzixin1/watchless`](https://github.com/chenzixin1/watchless)，评估基于提交 `a09c0edc6c3121c05e9824ef556256441d291d4f`。本文件是待确认方案，不改变现有产品定义或既有决策。

## 1. 结论

建议纳入，但只纳入 Watchless 的“视觉证据提取能力”，不直接把它的完整图文复刻产品接进课程，也不把现有深讲替换掉。

推荐形态是：把“图文精读”作为按内容条件启用的视觉学习能力，而不是每门课程固定出现的第三种模式。产品演示、工程教程、模型或论文解读中，只有当图表、代码、文档或界面状态本身承载关键证据时才生成；以口头观点为主、画面长期只是人物的访谈默认不生成。启用时，图文精读仍属于同一门课程，可以使用独立路由或页面，但读取同一份课程结构。

这样既符合已有“互动课程页是学习母体，图文、音频和视频从同一课程结构派生”的决策，也避免为了形态齐全而重复表达同一内容，或把 Watchless 接近完整转录的输出误当成教学内容。

## 2. 它解决的具体学习障碍

当前深讲已经有问题、作者判断、人话解释、机制、案例、工程价值和实践，但原视频中的图表、论文页面、界面状态和演示结果主要留在播放器里。用户第二遍复习时仍要在长视频和长文字之间来回找证据。

图文精读应解决三个障碍：

1. 读到观点时看不到对应的原片视觉证据；
2. 复习时必须重新播放视频，无法按章节快速扫描；
3. 工具演示、图表和论文解读只靠文字会损失关键状态与关系。

用户路径是：导读判断价值 → 系统判断原片画面是否承载学习证据 → 视觉型内容在难点模块或课后提供图文精读 → 点任一原片截图或引用回到对应时间戳 → 完成最小实践或闭卷复述。口头观点型内容不增加图文入口。

## 3. Watchless 的适配性

### 3.1 值得复用

| 能力 | 对本项目的价值 |
| --- | --- |
| `slides / explainer / conversation / demo` 四类内容路径 | 与工程教程、模型解读、产品评测、访谈观点四类验证视频有较好映射 |
| 语义切段而非固定间隔抽帧 | 可以围绕深讲模块中的完整论点、案例或操作步骤选择画面 |
| `document-evidence / screen-state / evidence` 等视觉策略 | 能优先保留论文图、数据图、代码/UI 完成状态，而非装饰性人物截图 |
| 候选帧接触表与最终帧人工/模型复核 | 适合建立“这张图是否真的帮助理解”的质量闸门 |
| `start_sec / end_sec / frame_timestamp_sec` | 可以与当前课程数据的模块和证据时间戳直接关联 |
| 转录覆盖、图片引用、重复帧审计 | 可借鉴为图文课程的可追溯与资产完整性检查 |

仓库使用 MIT License，代码层面允许复用和修改，但需要保留版权和许可声明。它目前是本地 Codex Skill，不是稳定的库或在线 API；仓库尚无 release，README 显示 13 次提交，因此不能把当前脚本结构视为稳定接口。

### 3.2 不应直接照搬

Watchless 的硬目标是“按原顺序完整保留，让读者不必再看原视频”。它生成近乎完整的 Light-plus 转录和逐场景图文解说。AI Video Course 的目标则是选出 2—3 个关键模块进行教学重组，并始终允许回到原片核对。两者内容合同不同。

以下部分不建议直接纳入首版：

- 完整视频逐场景图文复刻；
- Watchless 自带的 Markdown、Pandoc HTML、PDF 和 ZIP 页面模板；
- 默认火山引擎词级 ASR，因为当前课程已经有字幕输入，应先复用现有转录；
- 把 `video-use` Skill、Chrome、Pandoc、Poppler、火山 ASR 全部变成课程生成的硬依赖；
- 为所有访谈画面做完整说话人映射，除非该身份确实影响当前深讲模块；
- 默认公开发布原视频关键帧和完整图文产物。

后续已在临时隔离环境安装目标仓库依赖并实际运行测试，结果为 46 项通过。但使用真实 17 分 39 秒幻灯片视频 `eAXxdtNlK04` 时，默认 `hybrid-keyframe` 路径发生管道死锁：程序先持续读取 ffmpeg 的 stdout 原始帧，没有并发消费包含逐帧 `showinfo` 的 stderr，日志缓冲区写满后双方等待。`accurate` 回退路径可在约 4.6 秒内完成并识别 23 个幻灯片状态，但仍需要合并渐进动画产生的近重复状态。这证明单元测试通过不等于真实长视频管线可直接接入。

## 4. 页面方案比较

| 方案 | 优点 | 主要问题 | 结论 |
| --- | --- | --- | --- |
| 把现有深讲直接改成图文 | 改动入口少；读到文字就能看到证据 | 会破坏当前“播放器＋深讲＋时间戳”的互动学习母体；视觉密集视频会让课程页过长；无法清楚表达“第二遍复习” | 不推荐替换，可在现有深讲中只保留少量关键图预览 |
| 新建完全独立的图文课程页和生成内容 | 阅读专注，适合打印/PDF | 容易与互动课重复生成、内容漂移；用户不知道两套内容的关系；时间戳和学习状态难同步 | 不推荐成为独立产品 |
| 按内容启用“图文精读”，独立视图、共享数据 | 保留互动课主形态；只在视觉证据能提升理解时出现；内容和溯源只有一份 | 需要先判断视觉增益，并允许课程没有该入口 | **推荐** |

课程顶部不应固定展示三种形态。基础入口仍是互动课；重讲课只在已生成时出现；图文精读只在课程通过“视觉证据有增益”判断后出现。技术上图文精读可使用类似 `/course/<video_id>/reading` 的独立路由，返回互动课时保留当前模块位置；它不是新的课程实体。

建议启用判断：

- **启用**：产品/UI 演示、工程教程、代码执行、PPT/白板讲解、模型或论文图解、数据图表评测；
- **通常不启用**：棚内访谈、播客、圆桌、纯口播观点；
- **局部启用**：访谈中确实出现论文、报告、数据图或产品演示时，只为对应深讲模块生成视觉证据，不生成整期图文版。

## 5. 图文精读的信息结构

每个深讲模块按既有课程字段重新排版，不再生成另一套摘要：

1. 这部分解决什么问题；
2. 作者判断及其原片时间戳；
3. 原片视觉证据：图表、论文页、代码、界面状态或必要的人物/物体；
4. 人话解释；
5. 技术机制，必要时使用明确标注为“AI 解释”的机制图；
6. 原案例与失败边界；
7. 对用户项目的工程启示；
8. 15—30 分钟最小实践；
9. 闭卷复述提示。

原片截图与 AI 生成机制图必须视觉和标签分开：前者标记“原片画面＋时间戳”，后者标记“AI 解释图”，不能让生成图看起来像作者原有材料。每张原片图必须支持相邻的一条观点、案例或操作结果；纯装饰性人物截图不进入正文。

## 6. 数据与处理方案

现有 `course.json` 保持内容母体，只增加可选视觉证据资产。建议的最小字段形态：

```json
{
  "visual_evidence": [
    {
      "module_start_sec": 178,
      "frame_sec": 303,
      "image_path": "visuals/module-178/frame-303.jpg",
      "role": "chart|document|code|ui-state|speaker|object",
      "caption": "画面中可直接核对的事实，不补造作者观点",
      "supports": "author_judgment:0",
      "source_url": "https://youtu.be/VIDEO_ID?t=303"
    }
  ]
}
```

推荐处理链路：

1. 现有课程管线先用字幕生成知识地图和 2—3 个深讲模块；
2. 视觉适配器只处理这些深讲时间范围，不默认复刻整条视频；
3. 借鉴 Watchless 的视频路径、候选帧召回和证据画面策略，为每个“教学意图”选择关键帧；
4. 生成 `visual_evidence.json` 或合并为 `course.json` 的可选字段；
5. 校验帧时间位于模块范围、图片存在、无重复图、每张图有被支持的课程字段；
6. 图文精读视图只做确定性渲染，不重新调用模型改写课程内容。

没有授权的视频文件、无法可靠获取画面或关键帧无学习价值时，互动课仍应正常生成；图文精读显示“本模块没有可用的原片视觉证据”，不得使用无关封面或 AI 补画伪装来源证据。

## 7. 集成策略

不建议现在把 Watchless 仓库整体复制进项目，也不建议用 Git submodule 形成运行时强耦合。

更稳妥的顺序是：

1. **适配试验**：用固定提交 `a09c0ed...` 在项目外运行，验证其关键帧策略是否对当前课程有效；
2. **最小移植**：试验通过后，只移植或重写路径判断、候选帧抽取、关键帧选择与审计所需的最小代码，并保留 MIT 许可与来源说明；
3. **项目化接口**：输入固定为 `course.json + 已获授权的视频文件/来源`，输出固定为视觉证据 manifest，不依赖 Codex 手工写 `scene_NNN.md`；
4. **导出后置**：HTML/PDF 导出等图文阅读价值被验证后再做，不作为首版生成前提。

## 8. 风险与边界

- **版权与平台条款**：截图和完整图文复刻可能构成复制或改编。私人腾讯云站点与公开 GitHub Pages 必须分开处理；未获授权的关键帧不得默认提交公开仓库或公开部署。
- **隐私与第三方处理**：首版复用已有字幕，避免默认把音频发送给火山引擎。敏感或私人视频只允许本地处理。
- **生成成本**：Watchless 现有流程包含人工/Codex 路由、边界和选帧步骤，不是无人值守 API；必须记录实际耗时、候选帧数量和模型用量后再判断自动化程度。
- **信息过载**：Watchless 强调完整覆盖，本项目强调关键机制。选帧单位必须是“教学意图”，不是“每个场景都放一张图”。
- **维护性**：目标仓库没有稳定 release，首版必须锁定提交并通过本项目自己的 schema 和测试隔离变化。

## 9. 验证计划与通过标准

先用当前已有课程做一次窄范围技术试验，只处理两个深讲模块，验证数据接合、时间戳跳转和阅读布局。它只能证明适配器可行，不能证明产品价值。

在把能力纳入 MVP 前，必须按项目规则验证四类真实视频：

| 视频类型 | Watchless 路径重点 | 必须证明的机制 |
| --- | --- | --- |
| 工程教程 | `slides/demo + screen-state` | 关键代码或操作前后状态比纯文字更容易复现 |
| 模型解读 | `explainer + document-evidence` | 论文图、架构图或数据证据帮助复述机制 |
| 产品评测 | `demo + evidence` | 界面状态和结果证据支持判断产品边界 |
| 访谈观点 | `conversation + evidence` | 默认不生成图文精读；只有资料画面确实帮助理解时局部启用，用它验证“拒绝生成”是否正确 |

每条至少检查：

- 学习后能闭卷复述关键机制，质量不低于当前互动课；
- 图文复习耗时显著低于重看对应原片，且课程总学习时间仍不超过原片的 1/2；
- 随机抽查 5 处观点或图片，均能在原片 ±30 秒内核对；
- 每张图都能回答“它证明或解释了什么”，无纯装饰图和误导性截帧；
- 用户能基于模块完成最小实践，并说明课程机制如何迁移到自己的项目。

## 10. 推荐的下一步

若用户确认本方案，下一步只做一门现有课程、两个深讲模块的“图文精读”原型：不改互动课内容、不接完整 ASR、不做 PDF、不公开发布第三方关键帧。先比较桌面和手机的第二遍阅读体验，再进入四类真实视频验证。
