# 项目收口审计（2026-07-17）

> 本文记录收口前发现与本轮处理结果，不是新的产品定义来源。产品定义仍以 `docs/需求与方案.md` 为准。审计完成后，其他相关会话已归档或确认非活动，当前状态统一见 `tasks/current.md`。

## 1. 结论

项目目前不是“功能不够”，而是“验证债、状态债和成本守门缺失”已经超过继续扩功能的收益。

建议立即进入收口期：保留资讯速听与视频课程两个独立 MVP，但暂停新的连接器、知识库、视觉方向、部署路线和视频形态扩展。先完成三件事：

1. 用真实对照实验分别证明资讯速听的判断效率、视频课程的理解效率；
2. 给所有模型调用增加调用前预算、内容缓存和完整用量账本；
3. 把多会话协作从“共享工作区追加日志”改为“独立 worktree、小提交、单一当前状态页”。

在上述门槛通过前，不能把“页面可用、音频可播、定时任务可跑”写成“产品价值已验证”。

### 产品重新判断

“先筛选，再深入”仍是合理方向，但现在只是概念上的连续漏斗：资讯页的 `deepen` 只是建议标签，没有把当前来源、视频 URL 或证据传给课程管线；课程首页也没有接收速听条目并创建课程的工作流。因此应把两层继续当作两个独立实验：

- 实验 A：资讯速听是否显著提高一批内容的判断效率，同时不漏掉必看内容；
- 实验 B：结构化课程是否显著提高一条长视频的机制理解与追溯效率；
- 只有 A、B 各自通过后，才做实验 C：从速听中的长视频一键进入课程，是否提高真正的深挖率。

这比继续补“讲深一点”、推荐、知识库或更多连接器更小，也更能回答产品是否成立。

## 3. 过期、重复和冲突内容

### 高优先级冲突

1. `docs/多源资讯速听模式方案.md` 仍写“待生成首个 20 条真实速听包、验证后再开发邮箱连接器”，但 Gmail 自动任务、TTS、部署和登录已经实现；验证门槛仍未完成。
2. 同一方案写一批 8—10 条、6—8 分钟，PRD 写单条 20—60 秒、复杂项最多 90 秒；当前真实 9 条批次约 11 分 32 秒。决策日志虽放宽到 15 分钟，正式方案和指标没有同步说明这是实验变体。
3. README 把项目描述为视频课程，并把当前阶段写成课程与重讲原型；实际近期主要工作已经转向资讯速听、Gmail 和私有部署。
4. 决策日志包含证书剩余天数、当前公网入口等易过期运行态。这类内容应进入运行手册或状态页，不应作为长期产品决策。
5. `tasks/todo.md` 和 `tasks/progress.md` 同时保存计划、过程、回退、最终结果和重复 Review；同一视觉方向、Obsidian 路线、HTTPS 路线在文件中出现多次相反结论，无法回答“现在是什么”。

### 处理原则

- PRD 只保留产品目标、用户、边界、成功指标；不写服务器和临时发布状态。
- 决策日志只保留仍有效的重要选择；被推翻的决策标记 superseded，并链接替代项，不删除历史。
- 方案文档每节只允许 `current / experiment / superseded` 三种状态；“已实现”与“已验证”分开。
- `tasks/current.md` 只放当前阶段、唯一下一门槛、活跃任务和文件所有者；`todo/progress` 归档后不再作为当前状态源。
- 并发会话的原始任务日志差异保存在本地忽略目录；仓库内只补稳定摘要，不继续合并重复或含临时运行态的过程记录。

## 4. Token 与持续运行成本审计（修复前发现）

### P0：资讯任务先生成全部候选，最后才截取

`run_job` 对拉取到的每封邮件调用一次模型；`build_batch` 随后才按优先级和事件聚类截取 `max_items=10`。示例及本地配置允许 6 个发件人、每个最多 20 封、每封最多 50,000 字符：理论上单次可发生 120 次模型调用、最多输入 600 万字符，最终仍只展示 10 条。

当前本地 mail-drop 有 39 条记录、原始正文约 618,467 字符，按 50,000 字符上限裁剪后仍约 585,749 字符。粗略按每 Token 2—4 个字符估算，仅正文就是约 15 万—29 万输入 Token；这不是供应商账单，只用于说明调用前缺少预算门槛。文件中还有重复主题，说明模型调用前的候选去重也不充分。

### P0：用量账本低估且实际样本缺失

- JSON/schema 校验失败时会把完整 prompt 加上错误信息再调用一次，但只返回成功那次的 usage；第一次失败调用的 Token 没有累计。
- usage 只在整个批次成功并写出后落盘；中途失败的已消费 Token 不会留下账本。
- 当前 `local-data` 中没有找到任何 `usage.json`，课程与中文翻译管线也完全不记录 usage。
- 没有调用前的预计 Token、最大模型调用数、最大批次 Token 或金额预算；配置限制的是单封字符和最终条数，不是总成本。

### P0：缺少跨批次幂等缓存

- Gmail 只在本次抓取结果内按 message ID 去重，没有持久化“已处理 source/content hash”。24 小时回看窗口可能在相邻日重复处理同一封邮件。
- 资讯 TTS 只检查目标文件名是否存在，不校验文字、声线和参数 hash；同一目录内文字改变可能错误复用旧音频，新批次目录则会重新生成相同内容。
- 课程管线每次都把整份字幕发给模型；已有同视频、同字幕、同 prompt 版本的 `course.json` 也不会默认跳过。
- 中文翻译只支持整份 manifest 命中；一个 cue 或时间轴变化会导致整个片段重新翻译，且没有批次级 usage。

### P1：输出上限与内容粒度过宽

- 单课 `max_output_tokens=30000`，但没有真实输出分布支持该上限。
- 中文翻译每 35 cue 允许 8,000 输出 Token，也没有实际 usage 与输出长度统计。
- 邮件每封最多 5,000 输出 Token，并允许最多 120 封；最终丢弃的候选仍消耗输出费用。

### 推荐的成本守门顺序

1. **调用前筛选**：先做确定性垃圾邮件/验证码过滤、message ID 与内容 hash 去重、时间和发件人排序，再限制进入模型的邮件数；不要先为全部邮件生成候选。
2. **硬预算**：新增 `max_model_calls_per_run`、`max_total_input_chars`、`max_estimated_input_tokens` 和 `max_estimated_output_tokens`；超限时在调用前失败并说明被截断范围。
3. **持久缓存**：缓存键至少包含来源 ID、正文 hash、模型、prompt/schema 版本；相同输入直接复用结构化结果。课程与翻译采用同样规则。
4. **完整账本**：每次尝试结束立即追加 usage，包含成功/失败、重试、缓存命中、模型、prompt 版本、输入/输出 Token；批次失败也保留记录。
5. **TTS 内容寻址**：缓存键包含脚本文字、voice、rate、pitch；音频文件名或旁路 manifest 保存 hash，避免陈旧复用和跨批次重复生成。
6. **用数据收紧上限**：先记录 3—5 次真实运行，再根据 P95 输出收紧 30,000/8,000/5,000 上限，不先拍脑袋定价格。

建议初始安全门槛不是“尽量处理全部”，而是“每天最多选择 10 封进入模型、最多 10 次调用、总输入字符先限制在可人工检查的范围”；具体数值在首批 usage 样本后再调整。

## 5. 多会话协作风险与规则

### 当前风险

- 多个会话共享 `main` 工作区，现有 28 个已跟踪文件修改及大量未跟踪音频/历史文件，无法从 Git 判断所有权。
- `tasks/todo.md` 和 `tasks/progress.md` 是高冲突追加热点；重复内容已经超过其作为状态工具的价值。
- 生成音频和默认 briefing 数据进入 `prototype/`，容易把真实内容实验、公共 fixture 和产品代码混在同一提交。
- `.claude/launch.json` 等本机运行配置也在共同变更，容易制造与产品无关的冲突。

### 建议规则

1. 每个会话使用独立 `codex/<task>` 分支和独立 Git worktree；禁止多个会话直接写同一个 worktree。
2. 开工时在短状态页登记任务名、分支、worktree、文件所有者和预计交付；同一文件同一时间只能有一个 owner。
3. 一个提交只完成一个可验证目标；代码、测试、文档状态一起提交，生成二进制资产单独提交或保持本地。
4. 真实私人内容、每日音频、历史记录继续只放 `local-data/`；`prototype/` 只保留经选择的公开 fixture 和必要演示资产。
5. 合并前由主会话执行 `git diff --check`、完整单测和关键浏览器回归；不要让每个会话反复追加同一份全局 Review。
6. `tasks/current.md` 是唯一当前状态；完成后删除活跃项并链接 commit/审计证据，详细历史交给 Git，而不是无限追加 Markdown。

## 6. 收口路线与放行门槛

### 阶段 0：先停止成本失控

- 暂不启用每日自动任务；若已经启用，先保持手工触发。
- 为 Gmail 管线增加调用前筛选、硬预算、持久缓存和完整 usage。
- 为课程与中文翻译增加 usage 和同输入跳过机制。

放行标准：离线测试能证明超预算时零模型调用、重复输入零模型调用、重试用量完整累计、失败批次仍保存账本。

### 阶段 1：证明资讯速听

- 使用已经准备的 20 条素材完成裸读/裸看基线；不要再换样例或调整页面。
- 用同批内容完成速听判断、最终完整消费和指标复算。
- 只允许修复影响保真、听懂或操作完成的问题。

放行标准：沿用 PRD 的速度、一致率、0 必看漏判、保真、4/5 复述和长听标准；如 11 分 32 秒实验变体更好，先记录数据再修改正式时长。

### 阶段 2：证明视频课程

- 补齐工程教程、模型解读、产品评测三类黄金测试；访谈样例也按协议重新打分。
- 每条记录裸看/课程耗时、闭卷机制复述、5 处追溯和实际 Token。
- 暂不继续重讲视频和完整中文音轨形态。

放行标准：至少四类各一条；课程总时长不超过原片 1/2，关键机制无错误，5 处追溯全部命中。

### 阶段 3：验证两层连接

- 只实现一个最小动作：资讯条目若来源是长视频，携带 URL 进入课程导读生成/已有课程；其他来源仍打开原文。
- 记录用户从速听选择深入后，是否真的开始并完成课程。

放行标准：连接解决真实深挖需求，而不是只多一个导航入口。
