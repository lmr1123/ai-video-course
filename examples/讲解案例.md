# 讲解案例

本文定义一节合格课程模块应该长什么样。本文是虚构的结构样例（模板）；首个真实视频样例见 [课程样例-openai-codex-访谈.md](课程样例-openai-codex-访谈.md)。

## 样例：Agent Memory 不等于保存全部聊天记录

### 课程位置

- 内容类型：核心概念
- 默认难度：进阶入门
- 建议讲解时长：4—6 分钟
- 学习目标：理解“历史记录、上下文和长期记忆”的区别，并能判断自己的 Agent 是否真的需要记忆系统。

### 30 秒理解

Agent Memory 不是把所有历史聊天重新塞给模型，而是从经历中选择值得保留的信息，在未来需要时准确取回。前者像把整间仓库搬到桌上，后者像维护一套会整理、会检索、也会遗忘的笔记系统。

### 作者观点

正式案例中必须放作者原话或准确归纳，并附原视频时间戳。如果作者只讲了存储而没讲遗忘，不能把“遗忘机制”冒充作者观点。

### 技术机制

一个最小长期记忆流程包含：触发写入、抽取信息、去重或更新、持久化、按当前任务检索、把结果放回上下文、根据效果修正。真正困难的不是数据库，而是什么值得写、何时取回以及错误记忆如何纠正。

### 案例

一个代码 Agent 连续协助维护同一项目。普通聊天历史记录了所有对话；有效记忆只保留“项目使用 pnpm”“接口错误统一返回 Problem Details”“用户不接受自动修改数据库结构”等未来仍会影响决策的信息。下一次任务开始时，系统只检索与当前修改相关的记忆。

这个案例证明的是选择性写入和相关性检索，不足以证明记忆长期准确；后者还需要更新、冲突处理和遗忘机制。

### 对 vibe coding 开发者的价值

不要一看到 Agent 忘事就先加向量数据库。先判断问题是当前提示缺少项目规则、单次上下文过长，还是确实需要跨会话保留稳定信息。很多个人项目用一份维护良好的 `AGENTS.md` 已经足够。

### 最小实践

从最近一次 AI 开发对话中选出十条信息，只保留未来三次任务仍可能影响决策的内容；为每条写出触发检索的场景。如果大多数信息只在当前任务有效，就不应进入长期记忆。

### 讲解视频脚本骨架

```text
00:00 问题：为什么保存全部聊天记录仍然会“忘事”
00:30 类比：仓库和整理过的笔记本
01:15 机制：写入、更新、检索、使用、遗忘
02:40 代码 Agent 案例
04:00 常见误区：把向量数据库等同于记忆
04:40 最小实践与总结
```

## 案例验收规则

每个正式案例都要能回答：它在证明什么机制；输入是什么；经历了哪些步骤；怎样判断好坏；输出是什么；它不能证明什么；用户如何在自己的项目中复用。
